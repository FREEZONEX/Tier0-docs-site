## 일일 점검

<Steps>
1. Run the following commands from the deployment package root directory:
    ```bash
    ./bin/verify.sh --site config/site.conf
    ./bin/smoke-business-emqx.sh --site config/site.conf
    ```
2. Confirm the following items:
    - The 3 Swarm nodes are `Ready`.
    - `APP VIP /healthz` and `/readyz` return `200`.
    - `DB VIP:5432` is accessible.
    - The PostgreSQL primary, standby, and witness topology is normal.
    - `backend`, `redis`, `emqx`, `sourceflow`, `eventflow`, `marimo`, `opcua-server`가 모두 활성 애플리케이션 노드에서 실행 중입니다.
</Steps>
## 애플리케이션 점검
<Steps>
1. Run the following commands on the active application node:
    ```bash
    cd /opt/tier0-enterprise/tier0-deploy
    bash bin/compose.sh ps
    bash bin/compose.sh logs --tail=200 backend
    curl -fsS http://127.0.0.1:8088/healthz
    curl -fsS http://127.0.0.1:8088/readyz
    ```
    :::note
    `bin/compose.sh`를 우회해 raw `docker compose` command를 직접 실행하지 마세요. wrapper script는 installer가 생성한 runtime environment를 로드합니다.
    :::
2. Health-check interpretation:
    - `healthz` failure: check the backend process, image, and container logs first.
    - `readyz` failure: also check DB VIP, Redis, and database migration logs.
    - home page는 접근되지만 login이 불가능한 경우: `.env`의 `ADMIN_INITIAL_PASSWORD`를 확인한 뒤 backend IAM migration log와 user table을 확인하세요. HTTP port만으로 deployment 성공을 판단하지 마세요.
</Steps>
## 프로토콜入口 점검

```bash
nc -vz <APP_VIP> 1883
nc -vz <APP_VIP> 8883
nc -vz <APP_VIP> 8083
nc -vz <APP_VIP> 8084
nc -vz <APP_VIP> 4840
```
:::note
EMQX와 OPC UA Server는 활성 애플리케이션 노드에 고정됩니다. APP VIP 장애 조치는 `APP VIP:80` HTTP 진입점만 보장합니다. 프로토콜 포트는 HAProxy TCP 프록시로 라우팅되지 않으므로, 이 설계는 프로토콜 진입점의 고가용성을 보장하지 않습니다.
:::
## 데이터베이스 점검

Run the following on database nodes:

```bash
systemctl status tier0-postgresql
systemctl status tier0-repmgrd
sudo -iu postgres psql -Atqc 'select pg_is_in_recovery();'
sudo -iu postgres /usr/local/bin/repmgr -f /etc/repmgr.conf cluster show
```

- `pg_is_in_recovery=false` means primary.
- `pg_is_in_recovery=true` means standby.
- DB VIP must only be located on the current primary.
- standby replication이 중단되거나 witness가 upstream error를 보고하면 먼저 site를 보존하세요. full deployment를 직접 다시 실행하지 마세요.

## APP VIP Failover
<Steps>
1. Before testing, record the node that currently owns the VIP and verify:

    ```bash
    curl -fsS http://<APP_VIP>/healthz
    curl -fsS http://<APP_VIP>/readyz
    ```

2. 현재 VIP 노드에서 `keepalived`를 중지한 뒤 다른 애플리케이션 노드가 VIP를 인계했는지 확인하고 두 URL을 다시 테스트합니다. HAProxy 백엔드는 활성 애플리케이션 노드의 `8088/TCP`를 가리키므로, 세 애플리케이션 노드 모두 활성 노드의 8088 포트에 도달할 수 있어야 합니다.

3. `keepalived`를 복구한 뒤 `verify.sh`와 smoke check를 다시 실행합니다. drill 중에는 active application container를 중지하지 마세요.
</Steps>
## DB Primary 장애 전환

Before a failure drill, create a backup and confirm replication is healthy:

```bash
./bin/backup-db.sh --site config/site.conf
```

현재 primary에서 `tier0-postgresql`을 중지한 뒤 `repmgr`가 standby를 승격하고 DB VIP가 takeover하는지 확인합니다. 목표 RTO는 2분 이내입니다. 같은 architecture의 과거 테스트는 약 59-64초였습니다.
:::caution
- 이전 primary를 production topology에 직접 다시 시작하면 안 됩니다. 먼저 PostgreSQL, repmgrd, keepalived를 중지하고 남아 있는 DB VIP를 제거한 뒤, 현재 새 primary에서 basebackup을 실행하고 해당 node를 standby로 등록한 다음 witness를 다시 연결하세요.
- failover 후 full installer를 직접 다시 실행하지 마세요. inventory의 static primary/standby definition이 현재 topology를 교란할 수 있습니다.
:::
## 백업 및 복구

- Backup:

    ```bash
    ./bin/backup-db.sh --site config/site.conf
    ```

- Restore:

    ```bash
    ./bin/restore-db.sh --site config/site.conf --restore-file /backup/tier0-db/postgres-YYYYmmddTHHMMSS.dump
    ```
:::note
복원 전에 현재 primary를 확인하세요. 복원 후에는 `verify.sh`, 스모크 체크, 기본 사용자 경로의 실제 플랫폼 로그인 검증을 실행합니다.
:::
## 정리 및 재테스트

```bash
./full-cleanup.sh --env install.env --yes
./install.sh --auto --yes
```
:::caution
Full cleanup은 6개 노드의 Tier0 runtime 데이터, Docker/Swarm 데이터, 데이터베이스 백업을 삭제합니다. control container가 종료된 뒤 control node의 최종 정리도 수행합니다. 이 항목은 폐기 가능한 것으로 확인된 테스트 환경에서만 사용하세요. 고객 데이터나 incident 현장을 보존해야 한다면 일반 `--cleanup`을 사용하고 먼저 백업을 완료하세요.
:::
## 일반적인 문제

### Backend 시작 실패

```bash
cd /opt/tier0-enterprise/tier0-deploy
bash bin/compose.sh logs --tail=300 backend
bash bin/compose.sh ps
```
:::note
Focus on DB URL, DB VIP, Redis, migration, and mounted directory permissions.
:::
### EMQX 시작 실패

```bash
cd /opt/tier0-enterprise/tier0-deploy
bash bin/compose.sh logs --tail=200 emqx
```
:::note
Check certificates, configuration files, volume permissions, and whether ports `1883/8883/8083/8084` are occupied.
:::
### repmgrd 반복 재시작

```bash
systemctl status tier0-repmgrd
tail -100 /var/log/repmgr/repmgr.log
sudo -iu postgres psql -Atqc 'show shared_preload_libraries;'
```
:::note
`shared_preload_libraries`에 `repmgr`가 포함되어 있고 systemd unit이 `repmgrd --daemonize=false`를 사용하는지 확인하세요.
:::
### 자료 검증 실패

```bash
./bin/verify-materials.sh
```
:::caution
SHA256 verification을 건너뛰고 deployment를 계속하지 마세요. delivery package를 다시 받거나 traceable process에 따라 material을 다시 build하세요.
:::
