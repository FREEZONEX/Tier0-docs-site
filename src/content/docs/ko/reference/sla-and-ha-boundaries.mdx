이 문서는 6-VM 완전 오프라인 배포에 적용됩니다. 구성은 application / Swarm node 3개, PostgreSQL HA node 2개, witness node 1개입니다.

## 아키텍처 기준

| 계층 | 현재 설계 | 고가용성 경계 |
| --- | --- | --- |
| 애플리케이션 런타임 | Enterprise New compose는 고정된 `tier0_app_active_node`에서 실행됩니다. | `backend`, `redis`, `EMQX`, `SourceFlow`, `EventFlow`, `Marimo`, `OPC UA Server`는 아직 자동 multi-replica service가 아닙니다. |
| HTTP 진입점 | 세 애플리케이션 노드가 keepalived APP VIP + HAProxy를 실행합니다. | VIP는 이동할 수 있습니다. HAProxy는 활성 노드의 `8088/TCP`로 전달하고 `/healthz`를 확인합니다. |
| MQTT / OPC UA | EMQX는 `1883/8883/8083/8084`를 사용하고 OPC UA는 `4840`을 사용합니다. | 이들은 active node에 직접 연결됩니다. TCP VIP proxying에 포함되지 않으므로 protocol-entry HA는 보장되지 않습니다. |
| 데이터베이스 | PostgreSQL / TimescaleDB streaming replication + repmgr + witness + DB VIP. | primary 장애 시 standby를 자동 승격할 수 있습니다. asynchronous replication은 엄격한 zero RPO를 보장할 수 없습니다. |
| 백업 및 복원 | `backup-db.sh` / `restore-db.sh`. | 복구 가능성을 제공하지만 온라인 HA와 동일하지는 않습니다. |

## 권장 SLA

| 서비스 대상 | 권장 문구 | 제약 |
| --- | --- | --- |
| 플랫폼 HTTP 진입점 | 월 가용성 `99.5%`부터 시작합니다. | VIP failover는 active application node의 automatic migration과 같지 않습니다. |
| 데이터베이스 접근 | 운영 목표: `RTO <= 2 minutes`. | RPO는 replication lag에 따라 달라집니다. 같은 architecture의 과거 failover test는 약 59-64초였습니다. |
| MQTT / OPC UA | 현재 설계에서는 HA SLA를 약속하지 않습니다. | 먼저 HAProxy TCP proxying, 별도 VIP 또는 서비스 클러스터링을 추가해야 합니다. |
| 백업 및 복원 | 일일 백업, 주요 변경 전 백업 필수 | 복원 시간은 데이터 볼륨과 현장 I/O에 따라 달라집니다. |

`99.9%` 이상을 주장하기 전에 애플리케이션 자동 이전 또는 다중 복제본 배포, 프로토콜 진입점 고가용성, 모니터링 알림, 복구 훈련을 완료하세요. 계획된 역량을 이미 제공된 역량처럼 설명하지 마세요.

## 수락 기준

각 배포 또는 업그레이드는 최소한 다음 조건을 만족해야 합니다.

- `verify-materials.sh`가 통과하고 delivery package SHA256 검사가 통과해야 합니다.
- 세 Swarm 노드가 `Ready` 상태여야 합니다.
- `APP VIP /healthz`와 `/readyz`가 `200`을 반환하고 home page와 `/uns`에 접근할 수 있어야 합니다.
- 활성 노드의 7개 compose 서비스가 실행 중이어야 합니다.
- `DB VIP:5432`에 접근할 수 있고 primary, standby, witness topology가 정상이어야 합니다.
- 기본 계정이 실제로 로그인할 수 있어야 합니다. 포트, 컨테이너, HTTP 200 응답만으로 IAM 초기화 성공 여부를 판단하지 마세요.
- HTTP, EMQX 포트, WebSocket/WSS, OPC UA TCP smoke check가 통과해야 합니다.

## 점검 및 알림

매일 다음을 실행합니다.

```bash
./bin/verify.sh --site config/site.conf
./bin/smoke-business-emqx.sh --site config/site.conf
```

최소한 다음 알림을 구성합니다.

- APP VIP `/healthz` 또는 `/readyz` 실패.
- DB VIP 접근 불가, primary 없음, split-brain 위험, 복제 중단 또는 임계값을 초과한 복제 지연.
- keepalived, HAProxy, PostgreSQL 또는 repmgrd 비정상 상태.
- `backend`, `redis`, `EMQX` 등 핵심 컨테이너 종료 또는 빈번한 재시작.
- 데이터 디스크 사용량이 `80% / 90%` 초과.
- 최신 백업 실패 또는 미생성.

## 훈련 요구 사항

| 훈련 | 빈도 | 검증 기준 |
| --- | --- | --- |
| APP VIP failover | 납품 전 및 분기별 | 현재 VIP node에서 keepalived를 중지한 뒤 `/healthz`와 `/readyz`가 복구됩니다. |
| DB primary 장애 | 납품 전 및 6개월마다 | standby가 승격되고 DB VIP가 복구되며 이전 primary는 standby로 rebuild됩니다. |
| 데이터베이스 복원 | 매월 샘플 및 주요 변경 전 | 백업은 복원 가능해야 합니다. 복원 후 `verify`, 스모크 체크, 로그인 검증을 통과해야 합니다. |
| 자료 검증 | 모든 패키지 빌드 | source material, package SHA, 압축 해제 검증을 통과해야 합니다. |

## 사고 처리 원칙

1. 먼저 HTTP, MQTT, OPC UA, database, 단일 노드 영향 범위를 확인합니다.
2. VIP, HAProxy, 활성 애플리케이션, database primary 복구를 우선합니다.
3. 로그, 데이터 디렉터리, 백업을 보존하세요. incident 현장을 직접 정리하거나 덮어쓰지 마세요.
4. database incident에서는 먼저 현재 primary를 확인한 뒤 fail over, rejoin, restore 중 무엇을 할지 결정합니다.
5. 복구 후 `verify`, smoke check, 실제 로그인 검증을 실행합니다.

## 고 SLA 강화 항목

- 애플리케이션을 다중 복제본으로 구성하거나 활성 노드에 대한 검증 가능한 자동 이전 프로세스를 마련하세요.
- EMQX를 클러스터링하거나 전용 VIP가 있는 HAProxy TCP / TLS를 통해 라우팅합니다.
- HAProxy TCP, 전용 VIP 또는 서비스 클러스터링을 통해 OPC UA 고가용성 진입점을 제공합니다.
- PostgreSQL synchronous replication을 사용하거나 허용 가능한 replication lag와 RPO를 명확히 정의합니다.
- 백업 일정, off-site replication, 정기 restore drill을 자동화합니다.
- 컨테이너, 호스트, 데이터베이스, 로그, black-box probe를 통합 알림에 연결합니다.
