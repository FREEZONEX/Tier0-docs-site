## 계획 목표

이 계획은 offline 또는 private deployment environment에 하나의 Tier0 Enterprise 환경을 배포합니다. 총 6개의 virtual machine을 사용합니다.

- 애플리케이션 노드 3대: `Docker Swarm` 클러스터 기반을 구성하고 단일 복제본 Tier0 Enterprise 서비스와 상태 저장 **EMQX** 서비스를 실행합니다.
- 2 database nodes: form the `PostgreSQL + TimescaleDB` primary-standby cluster.
- 1 `Witness` node: participates in database monitoring, arbitration, and failover.

:::note[선택 사항]
- external `NFS/NAS` shared storage를 추가하고 세 Swarm node에 mount할 수 있습니다. Tier0 Enterprise의 node 간 shared file에 사용됩니다.
- This shared storage is an external dependency and is not counted as one of the 6 business VMs in this plan.
:::

:::tip[데이터베이스 리소스가 변경되는 경우]
이 계획은 각 database VM이 `4C / 8G / 500G`라고 가정합니다. 사용자 요구사항이 전체 database cluster 기준 `4C / 8G / 500G`라면, 구현 전에 performance와 capacity를 다시 계산해야 합니다.
:::

## 전체 토폴로지

<img class="t0-topology" src="/images/enterprise-ha-topology.png" alt="tier0-enterprise 6대 VM 남북향 데이터 경로" />

:::note
The diagram only shows the core links.
:::

- 비즈니스 접근 경로: 사용자 / 현장 시스템 -> `APP VIP` -> 세 개의 `Docker Swarm` 노드 -> `DB VIP` -> 현재 primary 데이터베이스.
- 토폴로지는 요청 진입점, 애플리케이션 처리, 데이터베이스 접근, 고가용성 모니터링을 위에서 아래로 보여주도록 배치되어 있습니다.
- 토폴로지 다이어그램에는 내부 클러스터 포트, SSH 운영 링크, 모니터링 링크가 표시되지 않습니다. 해당 내용은 <a href="#connection-list">연결 목록</a>과 <a href="#detailed-port-list">상세 포트 목록</a>에 설명되어 있습니다.
- `Witness`는 데이터베이스 arbitration, primary selection, health check에만 참여하며 비즈니스 접근 경로에는 포함되지 않습니다.
- `APP VIP` distributes traffic to the three Swarm nodes. The Swarm nodes access the database through `DB VIP`.
- `APP VIP`는 세 Swarm node 사이에서 fail over할 수 있습니다. `DB VIP`는 현재 PostgreSQL Primary role을 따르며 두 database node 사이에서 fail over할 수 있습니다.
- Tier0 Enterprise는 single-replica service로 실행됩니다. 현재 node가 실패하면 Swarm이 다른 healthy node에서 reschedule 및 restart합니다.
- `EMQX`는 상태 저장 서비스입니다. 일반적인 무상태 서비스처럼 임의로 재스케줄하면 안 됩니다. 기본적으로 영구 스토리지가 있는 하나의 노드에 고정해야 합니다. 메시징 계층에 고가용성이 필요하면 3노드 EMQX 클러스터를 배포하세요.
- optional `NFS/NAS`는 Swarm node용 shared file storage로만 사용되며 main database request path의 일부가 아닙니다.

## 데이터베이스 모니터링 및 Failover

#### Database failover path:

  <thead>
    <tr>
      <th>Stage</th>
      <th>Link</th>
      <th>Description</th>
    </tr>
  </thead>

    <tr>

      <td>Tier0 Enterprise replica on a Swarm node -&gt; <code>DB VIP</code> -&gt; current primary database</td>
      <td>The application always connects only to <code>10.60.20.10:5432</code>.</td>
    </tr>
    <tr>
      <td>Normal replication</td>

    </tr>
    <tr>
      <td>Normal monitoring</td>

    </tr>
    <tr>

      <td>Standby database -&gt; <code>Witness</code></td>
      <td>The standby database requests arbitration to avoid split-brain.</td>
    </tr>
    <tr>
      <td>Standby promotion</td>
      <td>Standby database <code>promote</code></td>
      <td>After <code>Witness</code> confirms the failover, the standby database is promoted to the new primary database.</td>
    </tr>
    <tr>

    </tr>
    <tr>
      <td>Application recovery</td>
      <td>Tier0 Enterprise replica on a Swarm node -&gt; <code>DB VIP</code> -&gt; new primary database</td>
      <td>The application connection pool reconnects, and read/write access is restored.</td>
    </tr>
  </tbody>
</table>

#### Key boundaries:

- `Witness`는 비즈니스 데이터를 저장하지 않고, 비즈니스 읽기/쓰기 트래픽을 처리하지 않으며, 애플리케이션에 데이터베이스 접근을 노출하지 않습니다.
- Business services only connect to `DB VIP:5432`. Direct connections to `ent-db-01/02` are not allowed.
- `DB VIP`는 현재 PostgreSQL Primary node에 binding되어야 합니다. database process 생존 여부만으로 fail over해서는 안 됩니다.
- `TimescaleDB`는 두 데이터베이스 노드 모두에서 PostgreSQL extension으로 배포됩니다. 독립 데이터베이스 인스턴스로 분리되지 않습니다.
- The recommended design is `PostgreSQL streaming replication + repmgr + repmgr witness + keepalived DB VIP`.
- 자동 장애 조치는 primary 데이터베이스 사용 불가 상황만 처리합니다. 원래 primary 데이터베이스가 복구되면 먼저 standby 데이터베이스로 다시 합류해야 하며, 자동으로 primary 역할을 되찾아서는 안 됩니다.

**Key rules:**

- Tier0 Enterprise can run as a single replica. Swarm only provides the cluster base and failover rescheduling capability.
- `EMQX`는 세션, 구독, 보관 메시지, 룰 엔진 상태, 영구 디렉터리를 가지므로 Tier0 Enterprise와 같은 임의 단일 복제본 재스케줄링 전략을 사용할 수 없습니다.
- `EMQX`를 단일 노드로 배포한다면 메시징 계층의 단일 장애점으로 취급하고 영구 디렉터리가 있는 하나의 Swarm 노드에 고정해야 합니다.
- 비즈니스가 `EMQX`의 단일 장애점을 허용할 수 없다면, 세 Swarm 노드에 3노드 EMQX 클러스터를 배포하세요. 각 노드는 고정 호스트 이름, 노드 이름, 영구 디렉터리를 가져야 합니다.

### NFS/NAS 공유 스토리지 규칙

NFS는 Tier0 Enterprise 단일 복제본이 Swarm 노드 사이에서 재스케줄된 뒤에도 같은 비즈니스 파일에 접근할 수 있도록 도입됩니다. 데이터베이스 primary-standby 복제나 상태 저장 EMQX 서비스의 경계를 바꾸지는 않습니다.

**Suitable content for NFS:**

- User-uploaded attachments
- Import/export files
- Report files
- Offline packages, templates, and reproducible intermediate files
- Business static files that must be read across nodes

**Content not recommended for NFS:**

- PostgreSQL / TimescaleDB data directories and WAL directories
- EMQX core data directories, Mnesia data, and runtime state
- Docker Engine data directory `/var/lib/docker`
- 고빈도 데이터베이스형 쓰기, 잠금에 민감한 파일, 강한 일관성이 필요한 큐

**NFS boundaries:**

- NFS is a shared file dependency. It is not a database high availability component.
- NFS 자체가 단일 장애점이라면 파일 관련 기능도 단일 장애점을 갖습니다. 운영 환경에서는 기존 HA NAS 또는 dual-controller storage를 사용하세요.
- Tier0 Enterprise 단일 복제본이 다른 Swarm 노드로 재스케줄되기 전에 대상 노드는 NFS가 마운트되어 있고 읽기/쓰기 접근이 검증되어야 합니다.
- If Tier0 Enterprise does not write local files at all, NFS is not required.

## 서비스 배포 권장사항

### Docker Swarm

- Set all three application nodes as `manager + worker`.
- Deploy Tier0 Enterprise by using `docker stack deploy`.
- Tier0 Enterprise는 기본적으로 `replicas: 1`을 사용합니다. 노드 장애 후 Swarm은 이를 다른 정상 노드로 재스케줄합니다.
- Tier0 Enterprise containers do not store critical persistent data. Business data is stored in PostgreSQL/TimescaleDB.
- 각 Swarm 노드는 `100G` 스토리지만 가지고 이를 단일 시스템 디스크로 사용합니다. Docker 데이터 디렉터리는 시스템 디스크에 유지됩니다. 별도 데이터 디스크는 계획되어 있지 않습니다.
- Use `Docker config` for configuration files.
- Use `Docker secret` for passwords, database credentials, and certificate private keys.
- 배포 중 공용 네트워크 의존성을 피하기 위해 현장 내부 컨테이너 이미지 registry를 사용하세요.
- service를 publish할 때 health check를 구성해 준비되지 않은 container로 traffic이 전달되지 않도록 합니다.
- 애플리케이션이 노드 간 공유 파일을 필요로 한다면 NFS/NAS에서 공유 디렉터리를 마운트하세요. 중요한 파일을 로컬 임시 컨테이너 디렉터리에 의존하지 마세요.
- `/var/lib/docker`가 가득 차지 않도록 이미지, 컨테이너 로그, 빌드 캐시에 대한 정리 정책을 구성하세요.

**Single-replica deployment example:**

```yaml
deploy:
  replicas: 1
  restart_policy:
    condition: any
  update_config:
    order: stop-first
```

### EMQX Stateful 서비스

`EMQX`는 상태 저장 서비스로 취급해야 합니다. 일반적인 무상태 Swarm 복제본처럼 임의로 재스케줄하면 안 됩니다.

<table>
  <thead>
    <tr>
      <th>Mode</th>
      <th>Recommended Scenario</th>
      <th>Deployment Rule</th>
      <th>Risk</th>
    </tr>
  </thead>

    <tr>
      <td>Single-node fixed mode</td>

**Minimum requirements for single-node fixed mode:**

- Use a Swarm placement constraint to fix the node, for example `node.labels.emqx == true`.
- Fix the data directory to the host, for example `/data/emqx`.
- Include configurations, certificates, plugins, and rule engine configurations in backup.
- node가 실패하면 먼저 원래 node를 복구하세요. 복구할 수 없다면 backup에서 새 node로 복구합니다. stateless random failover처럼 처리하지 마세요.

**Minimum requirements for three-node cluster mode:**

- Fix one EMQX instance on each Swarm node.
- 각 EMQX node에 fixed node name을 사용하세요. container가 재생성되어도 node name이 바뀌면 안 됩니다.
- Open cluster communication ports only between the three Swarm nodes.
- Expose the Dashboard only to the O&M network segment.
- MQTT access는 `APP VIP`, 별도 MQTT VIP 또는 layer 4 load balancer를 통해 노출할 수 있습니다. 구현 전에 entry method를 확인하세요.
- EMQX 핵심 데이터 디렉터리를 NFS에 두지 마세요. 단일 노드 고정 모드에서는 고정 노드의 로컬 디스크를 사용하고 적절히 백업하세요. 3노드 모드에서는 EMQX 클러스터 메커니즘에 의존합니다.

### NFS/NAS Shared Storage

- NFS/NAS is an optional external dependency and does not count as one of the six VMs.
- Mount the same shared directory on all three Swarm nodes.
- Use it only for shared files that Tier0 Enterprise needs to access across nodes.
- Do not use it for PostgreSQL/TimescaleDB data directories.
- Do not use it for EMQX core runtime data.
- service가 NFS가 mount되지 않은 node로 이동한 뒤 local directory에 쓰지 않도록 pre-start mount check를 구성하세요.
- NFS가 실패해도 database read/write access는 직접 영향을 받지 않지만 upload, import, export, report 같은 기능은 실패할 수 있습니다.

### APP VIP

- Deploy `APP VIP` on the three Swarm nodes. Use `keepalived + HAProxy` or `keepalived + Nginx`.
- The VIP is only the unified entry. It does not require a separate VM.
- Health checks should use the application `/health` or `/ready` endpoint.
- If any Swarm node fails, the VIP moves to another healthy node.

### PostgreSQL + TimescaleDB

**Recommended baseline:**

- PostgreSQL: `16` or a version certified on site.
- TimescaleDB: choose a stable version that matches the PostgreSQL major version.
- Primary-standby replication: physical streaming replication.
- Failover: `repmgrd + witness`.
- Database entry: `DB VIP`.
- 연결 풀: 데이터베이스 연결이 과도해지지 않도록 애플리케이션 측 또는 별도 컨테이너에서 `PgBouncer`를 사용하는 것을 권장합니다.

### Witness

- `Witness` node는 `repmgr witness` metadata만 저장하며 database failure detection과 arbitration을 돕습니다.
- `Witness`는 비즈니스 데이터를 저장하지 않고, 비즈니스 읽기/쓰기 트래픽을 처리하지 않으며, 애플리케이션에 데이터베이스 접근을 노출하지 않습니다.
- `Witness`는 안정적인 네트워크 접근이 필요합니다. 높은 스토리지 성능은 필요하지 않습니다.
- `Witness` 장애가 현재 비즈니스 접근을 중단시키지는 않아야 하지만, 자동 장애 조치의 신뢰성을 낮춥니다.
- After `Witness` recovers, verify that it has rejoined the database cluster state.

## 백업, 모니터링, 점검

**Minimum delivery requirements:**

- Perform a full database backup every day and keep backups for at least `7` days.
- Enable WAL archiving or incremental backups to meet the on-site RPO.
- Perform a restore drill every month to verify that backups are usable.
- Swarm 서비스 헬스 체크, 컨테이너 재시작 횟수, 노드 디스크 사용량, 메모리, CPU를 모니터링합니다.
- EMQX 연결, 구독, 메시지 처리량, 큐 백로그, 룰 엔진 실패, 영구 디렉터리 사용량을 모니터링합니다.
- NFS 가용성, 마운트 상태, 용량, inode 사용량, 읽기/쓰기 지연 시간을 모니터링합니다.
- PostgreSQL primary-standby 지연, 복제 슬롯, WAL 백로그, 연결 수, 느린 SQL, 데이터베이스 bloat를 모니터링합니다.
- 정기 점검에 현재 VIP owner, repmgr 클러스터 상태, Witness 상태를 포함하세요.

## 구현 순서

1. Create 6 VMs and fix hostnames, IP addresses, time synchronization, and SSH access.
2. Initialize system baselines, firewall rules, disk mounts, and kernel parameters.
3. NFS/NAS를 사용한다면 먼저 shared directory setup, permission, mounting, restart auto-mount verification을 완료하세요.
4. Deploy the three-node Docker Swarm.
5. Deploy `APP VIP + HAProxy/Nginx`.
6. EMQX를 단일 노드 고정 모드 또는 3노드 클러스터 모드로 배포합니다.
7. Deploy PostgreSQL + TimescaleDB primary-standby.
8. Deploy repmgr witness, repmgrd, and the DB VIP failover script.
9. Publish the Tier0 Enterprise single-replica stack.
10. 연결성, NFS 읽기/쓰기 접근, EMQX 복구, 데이터베이스 장애 조치, 백업 복원, 정기 점검을 검증합니다.
11. Finalize operation documents, archive account credentials, and set up routine inspection tasks.

## 위험 및 확인 항목

- 실제 현장 네트워크 세그먼트, 게이트웨이, DNS, NTP, 인증서, 이미지 registry 주소를 확인해야 합니다.
- Tier0 Enterprise에 local file persistence가 필요한지는 아직 확인되지 않았습니다. 필요하다면 shared storage 또는 object storage solution을 추가하세요.
- NFS/NAS가 기존 현장 스토리지에서 제공되는지 확인해야 합니다. NFS가 단일 장애점이라면 공유 파일 기능도 단일 장애점을 갖습니다.
- 각 Swarm 노드에는 `100G` 단일 시스템 디스크만 있습니다. 이미지, 컨테이너 로그, 임시 파일 수를 관리하세요. 장기 비즈니스 데이터 저장에는 적합하지 않습니다.
- `EMQX`는 상태 저장 서비스입니다. 구현 전에 단일 노드 고정 모드를 사용할지 3노드 클러스터 모드를 사용할지 확인하세요. 단일 노드 고정 모드에서는 메시징 계층에 단일 장애점이 있습니다.
- Database RPO/RTO는 아직 확인되지 않았습니다. 현재 계획은 primary-standby automatic failover를 사용하며 zero data loss를 보장할 수 없습니다.
- `DB VIP` 장애 조치 스크립트는 현장에서 검증해야 하며, 특히 가상화 플랫폼이 VRRP를 허용하는지 확인해야 합니다.
- on-site network가 VRRP를 허용하지 않는다면 load balancer, DNS switch 또는 virtualization-platform floating IP를 대신 사용하세요.
