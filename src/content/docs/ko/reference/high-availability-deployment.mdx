## 계획 목표

이 계획은 오프라인 또는 프라이빗 환경에 Tier0 Enterprise 환경 하나를 배포하기 위한 것입니다. 기본 구성은 총 6대의 가상 머신입니다.

- 애플리케이션 노드 3대가 `Docker Swarm` 클러스터를 구성하고, 단일 replica Tier0 Enterprise 서비스와 stateful **EMQX** 서비스를 실행합니다.
- 데이터베이스 노드 2대가 `PostgreSQL + TimescaleDB` primary-standby 클러스터를 구성합니다.
- `Witness` 노드 1대가 데이터베이스 모니터링, 중재, failover에 참여합니다.

:::note[선택 사항]
외부 `NFS/NAS` 공유 스토리지를 추가하고 세 Swarm 노드에 마운트할 수 있습니다. 이는 Tier0 Enterprise의 노드 간 공유 파일용 외부 의존성이며, 이 계획의 6대 업무 VM에는 포함하지 않습니다.
:::

:::tip[데이터베이스 리소스가 변경되는 경우]
이 계획은 각 데이터베이스 VM이 `4C / 8G / 500G`라고 가정합니다. 전체 데이터베이스 클러스터 기준 요구사항이 `4C / 8G / 500G`라면, 구현 전에 성능과 용량을 다시 계산해야 합니다.
:::

## 전체 토폴로지

<img class="t0-topology" src="/images/enterprise-ha-topology.png" alt="tier0-enterprise 6 VM 남북 데이터 경로" />

그림은 핵심 링크만 보여줍니다.

- 업무 접근 경로: 사용자 / 현장 시스템 -> `APP VIP` -> 세 `Docker Swarm` 노드 -> `DB VIP` -> 현재 primary 데이터베이스.
- `APP VIP`는 세 Swarm 노드로 트래픽을 분산합니다. Swarm 노드는 `DB VIP`를 통해 데이터베이스에 접근합니다.
- `APP VIP`는 세 Swarm 노드 사이에서 failover할 수 있습니다. `DB VIP`는 현재 PostgreSQL Primary 역할을 따라가며 두 데이터베이스 노드 사이에서 failover할 수 있습니다.
- `Witness`는 데이터베이스 중재, primary 선택, health check에만 참여하며 업무 접근 경로에는 포함되지 않습니다.
- `EMQX`는 stateful 서비스입니다. 기본적으로 영구 스토리지가 있는 한 노드에 고정해야 합니다. 메시징 계층에도 HA가 필요하면 3노드 EMQX 클러스터를 배포하세요.

## 데이터베이스 모니터링 및 Failover

| 단계 | 링크 | 설명 |
| --- | --- | --- |
| 정상 접근 | Swarm 노드의 Tier0 Enterprise replica -> `DB VIP` -> 현재 primary | 애플리케이션은 항상 `10.60.20.10:5432`에만 연결합니다. |
| 정상 복제 | 현재 primary -> 현재 standby | PostgreSQL WAL streaming replication으로 데이터를 동기화합니다. |
| 정상 모니터링 | `DB-01` / `DB-02` -> `Witness` | `repmgrd`가 상태를 보고하고 `Witness`가 중재에 참여합니다. |
| primary 장애 | standby -> `Witness` | standby가 split-brain을 피하기 위해 중재를 요청합니다. |
| standby 승격 | standby `promote` | `Witness` 확인 후 standby가 새로운 primary로 승격됩니다. |
| VIP failover | `DB VIP` -> 새 primary | `10.60.20.10`이 새 primary에 바인딩됩니다. |
| 애플리케이션 복구 | Tier0 Enterprise -> `DB VIP` -> 새 primary | 연결 풀이 재연결되고 읽기/쓰기가 복구됩니다. |

핵심 경계:

- 업무 서비스는 `DB VIP:5432`에만 연결하며 `ent-db-01/02` 직접 연결은 허용하지 않습니다.
- `Witness`는 업무 데이터를 저장하지 않고, 업무 읽기/쓰기 트래픽을 처리하지 않으며, 애플리케이션에 데이터베이스 접근을 노출하지 않습니다.
- `TimescaleDB`는 두 데이터베이스 노드 모두에 PostgreSQL extension으로 배포됩니다.
- 권장 설계는 `PostgreSQL streaming replication + repmgr + repmgr witness + keepalived DB VIP`입니다.
- 자동 failover는 primary 데이터베이스가 사용할 수 없는 경우만 처리합니다. 원래 primary가 복구되면 먼저 standby로 다시 참여해야 합니다.

## 배포 권장사항

- `docker stack deploy`로 Tier0 Enterprise를 배포합니다.
- 설정 파일에는 `Docker config`를 사용하고, 비밀번호, 데이터베이스 자격 증명, 인증서 private key에는 `Docker secret`을 사용합니다.
- 배포 중 공용 네트워크 의존성을 피하려면 현장 내부 이미지 registry를 사용합니다.
- `APP VIP`는 세 Swarm 노드에 `keepalived + HAProxy` 또는 `keepalived + Nginx`로 배포합니다.
- 데이터베이스 진입점은 `DB VIP`로 통일합니다. 데이터베이스 연결 수가 과도해지지 않도록 애플리케이션 측 연결 풀 사용을 권장합니다.
- NFS/NAS는 Tier0 Enterprise의 노드 간 공유 파일에만 사용하고 PostgreSQL/TimescaleDB 데이터 디렉터리나 EMQX 핵심 runtime 데이터에는 사용하지 마세요.

## 포트 및 방화벽

전체 포트 세부 정보는 [표준 포트 목록](../standard-port-list/)을 참고하세요.

기본적으로 열지 않을 포트:

- Docker remote API `2375` / `2376`. 꼭 필요한 경우에도 O&M 네트워크로 제한하세요.
- 데이터베이스 노드의 `80/443`. 데이터베이스 노드는 Web 진입점을 호스팅하지 않습니다.
- 애플리케이션에서 Witness로의 `5432` 접근. Witness는 업무 SQL을 처리하지 않습니다.

## 고가용성 기능 및 장애 경계

| 장애 시나리오 | 시스템 동작 | 경계 |
| --- | --- | --- |
| 현재 Swarm 노드 장애 | Swarm이 단일 replica 서비스를 다른 정상 노드로 재스케줄합니다. | 컨테이너 재시작과 애플리케이션 재연결을 기다려야 합니다. |
| 단일 노드 고정 모드 EMQX 장애 | 먼저 원래 노드를 복구하고, 불가능하면 백업에서 새 노드로 복원합니다. | 메시징 계층이 중단되며 stateless 서비스처럼 무손실 failover할 수 없습니다. |
| 3노드 EMQX 클러스터의 단일 노드 장애 | 남은 노드가 계속 서비스하고 장애 노드는 복구 후 다시 참여합니다. | EMQX 클러스터 구성이 사전에 완료되어야 합니다. |
| 현재 primary 장애 | Witness 중재 후 standby가 승격되고 `DB VIP`가 failover합니다. | 쓰기가 수초에서 수분 중단되며 애플리케이션은 재연결해야 합니다. |
| standby 장애 | primary가 계속 서비스하고 복제 중단 알림이 발생합니다. | 자동 failover 능력이 감소합니다. |
| Witness 장애 | 현재 primary가 계속 서비스합니다. | 자동 failover 능력이 감소하므로 Witness를 먼저 복구하세요. |
| NFS/NAS 장애 | 공유 파일에 의존하는 기능이 영향을 받습니다. | 데이터베이스와 메모리 내 업무 프로세스는 반드시 중단되지는 않습니다. |

## 백업, 모니터링, 점검

- 매일 데이터베이스 백업을 실행하고, 큰 변경 전에는 반드시 백업합니다.
- APP VIP, DB VIP, PostgreSQL replication lag, Swarm 노드 상태, EMQX 상태, 디스크 사용률을 모니터링합니다.
- primary 전환, 애플리케이션 노드 장애, 백업 복원, 공유 스토리지 장애를 정기적으로 리허설합니다.
- 운영 전 SSH passwordless 접근, NTP 시간 동기화, 이미지 registry, 인증서, DNS, gateway, 방화벽 규칙을 확인합니다.

## 구현 순서

1. 현장 네트워크, VIP, DNS, NTP, 인증서, 이미지 registry, 방화벽 정책을 확인합니다.
2. 6대 VM의 OS, 디스크, 시간 동기화, 기본 보안 설정을 준비합니다.
3. NFS/NAS를 사용하는 경우 공유 디렉터리, 권한, 마운트, 재부팅 후 자동 마운트를 검증합니다.
4. 3노드 Docker Swarm을 배포합니다.
5. `APP VIP + HAProxy/Nginx`를 배포합니다.
6. 필요에 따라 단일 노드 고정 EMQX 또는 3노드 EMQX 클러스터를 배포합니다.
7. PostgreSQL + TimescaleDB primary-standby를 배포합니다.
8. repmgr witness, repmgrd, DB VIP failover script를 배포합니다.
9. Tier0 Enterprise 단일 replica stack을 게시합니다.

## 위험 및 확인 항목

- 현장 네트워크 세그먼트, gateway, DNS, NTP, 인증서, 이미지 registry 주소를 확인해야 합니다.
- Tier0 Enterprise가 로컬 파일 영속성을 필요로 하는지 확인하세요. 필요하면 공유 스토리지 또는 object storage 방안을 추가합니다.
- `DB VIP` failover script는 현장에서 검증해야 하며, 특히 가상화 플랫폼이 VRRP를 허용하는지 확인해야 합니다.
- 현장 네트워크가 VRRP를 허용하지 않는 경우 load balancer, DNS 전환 또는 가상화 플랫폼 floating IP를 사용하세요.
