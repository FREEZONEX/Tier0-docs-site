### 물리 데이터

물리적 위치가 있는 기계와 설비의 산업 데이터는 ISA-95 표준을 기반으로 모델링합니다.

#### ISA-95 Equipment Hierarchy
ISA-95 설비 계층은 조직 내 물리 자산과 제어 시스템이 어떻게 구조화되는지 설명합니다.
다음 계층으로 나뉩니다.
- **Enterprise**: 전체 조직을 나타내는 최상위 계층입니다. ERP system이 보통 지원하는 전체 business process와 decision-making entity를 포함합니다.  
- **Site**: 특정 지리적 위치나 시설을 나타냅니다. 각 site는 자체 management system과 local production control을 가질 수 있습니다. 
- **Area**: site 안의 운영 구역으로, packaging, assembly, warehousing 같은 기능 영역인 경우가 많습니다. 
- **Process Cell**: area 안의 제조 기본 단위입니다. 특정 제품을 생산하거나 특정 기능을 수행하는 데 필요한 모든 equipment를 포함합니다. 
- **Unit**: process cell의 독립된 부분으로, mixing, heating, assembling 같은 특정 기능을 담당합니다. 
- **Equipment Module**: unit 안에서 특정 작업을 수행하는 모듈형 장비입니다. 예를 들어 mixing unit에서는 heating element가 module이 될 수 있습니다.
- **Control Module**: 최하위 계층으로, 물리 공정을 직접 제어하는 sensor, pump, actuator, PID control loop 같은 개별 device나 controller를 나타냅니다.

### 비즈니스 데이터

ERP의 생산 주문, 작업 지시 계획 같은 비즈니스 계획 및 프로세스 객체는 비즈니스 context에 맞게 모델링합니다.
:::tip[비즈니스 데이터를 식별하는 방법]
물리적 위치가 없는 데이터입니다.
:::
#### 비즈니스 데이터 모델링 Guide
```
Factory / Business Domain / Business Object / Topic Type / Topic
```
- **Business Domain**: Confirm the business process or system the data belongs to.
    ```
    ERP
    MES
    WMS
    QMS
    EAM
    LIMS
    Planning
    Inventory
    Maintenance
    Quality
    Logistics
    ...
    ```
- **Business Object**: The core managed object in the business domain.
    ```
    ProductionOrders
    WorkOrderPlan
    MaterialInventory
    QualityInspection
    MaintenanceOrders
    PurchaseOrders
    Shipments
    ...
    ```
- **Topic Type**: Choose the data type from **Metric**, **Action**, and **State**.

    | Topic Type | Used for | Examples |
    | --- | --- | --- |
| STATE | 현재 상태 또는 현재 사실 | work order status, order status, equipment status |
    | METRIC | 지속적으로 변하는 숫자 값 | Temperature, speed, output, OEE, energy consumption |
    | ACTION | 외부 실행자를 위한 command | 설비 시작, 설비 정지, alarm reset, work order dispatch 요청 |

## Topic 폭증을 피하는 방법
Tier0는 topic 폭증을 피하기 위해 `Metric`, `Action`, `State` 세 가지 고정 topic type을 사용합니다. 

- **Use `Metric` for high-frequency real-time data**

  각 signal 세부 항목마다 namespace를 확장하지 말고, 고빈도 측정값과 telemetry는 `Metric` 아래에 둡니다.

  <Tabs syncKey="topic-explosion-metric">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Machine_001
          ├── Temperature_Value
          ├── Temperature_Unit
          ├── Temperature_Status
          └── Speed_Value
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Metric
          └── Machine_001
  ```

  Model fields:

  ```json
  {
    "temperature": 80,
    "temperature_unit": "Celsius",
    "temperature_status": "Normal",
    "speed": 300
  }
  ```

    </TabItem>
  </Tabs>

- **Use `State` for low-frequency status data**

  저빈도 상태, snapshot, 현재 비즈니스 상태는 `State` 아래에 둡니다. 작업, batch, 주문처럼 변하는 entity마다 새 topic을 만들지 않습니다.

  <Tabs syncKey="topic-explosion-state">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── State
          ├── Job_001
          ├── Job_002
          └── Job_003
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── State
          └── Machine_001
  ```

  Model fields:

  ```json
  {
    "current_job": "Job_003",
    "product": "Product_A"
  }
  ```

    </TabItem>
  </Tabs>

- **Use `Action` for low-frequency action data**

  명령, 요청, operator action은 `Action` 아래에 둡니다. action 입력은 parameter별 topic이 아니라 topic의 field로 유지합니다.

  <Tabs syncKey="topic-explosion-action">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Machine_001
          ├── Start_Command
          ├── Target_Height
          └── Light_On
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Action
          └── StartJob
  ```

  Model fields:

  ```json
  {
    "target_height": 12.5,
    "light_on": true,
    "requested_by": "operator"
  }
  ```

    </TabItem>
  </Tabs>

## Payload 구조

topic payload를 설계할 때는 소비할 데이터와 다시 쓸 데이터를 모두 고려해야 합니다.

- **Action**

  Action payload에는 데이터가 Tier0 데이터베이스에 저장된 시간을 기록하는 `_timestamp`가 자동 포함됩니다.

  <Tabs syncKey="payload-action">
    <TabItem label="start_job">

  ```json
  {
    "light_turn_on": true,
    "height_adjust": 12.5,
    "_timestamp": "2026-07-10T10:00:00Z"
  }
  ```

    </TabItem>
    <TabItem label="stop_job">

  ```json
  {
    "water_pump_off": true,
    "window_close": true,
    "_timestamp": "2026-07-10T10:00:00Z"
  }
  ```

    </TabItem>
  </Tabs>

- **Metric**

  Metric payload에는 데이터가 Tier0 데이터베이스에 저장된 시간을 기록하는 `_timestamp`와 데이터 품질을 나타내는 `_quality`가 자동 포함됩니다.

  <Tabs syncKey="payload-metric">
    <TabItem label="ERP / SplitCount">

  ```json

    "production_order_id": "PO-20260710-001",
    "plan_id": "PLAN-001",
    "plan_number": "WOP-20260710-001",
    "split_count": 3,
    "updated_at": "2026-07-10T10:00:00Z",
    "_timestamp": "2026-07-10T10:00:00Z",
    "_quality": "Good"
  }
  ```

    </TabItem>
    <TabItem label="Site_01 / Target_Qty">

  ```json
  {
    "production_order_id": "PO-20260710-001",
  ```
