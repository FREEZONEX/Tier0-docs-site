## Plan Objective

This plan deploys one Tier0 Enterprise environment in an offline or private deployment environment. It uses 6 virtual machines in total:

- 3 application nodes: form the `Docker Swarm` cluster base and run the single-replica Tier0 Enterprise service and the stateful **EMQX** service.
- 2 database nodes: form the `PostgreSQL + TimescaleDB` primary-standby cluster.
- 1 `Witness` node: participates in database monitoring, arbitration, and failover.

:::note[Optional]
- An external `NFS/NAS` shared storage can be added and mounted by the three Swarm nodes. It is used for Tier0 Enterprise shared files across nodes.
- This shared storage is an external dependency and is not counted as one of the 6 business VMs in this plan.
:::

:::tip[When database resource changes]
This plan assumes that each database VM has `4C / 8G / 500G`. If the user requirement is `4C / 8G / 500G` for the whole database cluster, performance and capacity must be recalculated before implementation.
:::

## Overall Topology

<img width={650} src="/images/enterprise-ha-topology.png" />

:::note
The diagram only shows the core links.
:::

- Business access path: users / on-site systems -> `APP VIP` -> three `Docker Swarm` nodes -> `DB VIP` -> current primary database.
- The topology is arranged from top to bottom to show request entry, application processing, database access, and high availability monitoring.
- The topology diagram does not show internal cluster ports, SSH O&M links, or monitoring links. These are described in the <a href="#connection-list">Connection List</a> and <a href="#detailed-port-list">Detailed Port List</a>.
- `Witness` only participates in database arbitration, primary selection, and health checks. It is not in the business access path.
- `APP VIP` distributes traffic to the three Swarm nodes. The Swarm nodes access the database through `DB VIP`.
- `APP VIP` can fail over between the three Swarm nodes. `DB VIP` follows the current PostgreSQL Primary role and can fail over between the two database nodes.
- Tier0 Enterprise runs as a single-replica service. If the current node fails, Swarm reschedules and restarts it on another healthy node.
- `EMQX` is a stateful service. It should not be randomly rescheduled like a normal stateless service. By default, it should be fixed to one node with persistent storage. If high availability is required for the messaging layer, deploy a three-node EMQX cluster.
- The optional `NFS/NAS` is only used as shared file storage for the Swarm nodes. It is not part of the main database request path.

#### Database failover path:

      <td>Tier0 Enterprise replica on a Swarm node -&gt; <code>DB VIP</code> -&gt; current primary database</td>
      <td>The application always connects only to <code>10.60.20.10:5432</code>.</td>

    <tr>
      <td>Normal replication</td>
      <td>Current primary database -&gt; current standby database</td>
      <td>Data is synchronized through PostgreSQL WAL streaming replication.</td>

    </tr>
    <tr>

      <td>Standby database -&gt; <code>Witness</code></td>
      <td>The standby database requests arbitration to avoid split-brain.</td>
    </tr>
    <tr>
      <td>Standby promotion</td>
      <td>Standby database <code>promote</code></td>
      <td>After <code>Witness</code> confirms the failover, the standby database is promoted to the new primary database.</td>
    </tr>
    <tr>
      <td>VIP failover</td>
      <td><code>DB VIP</code> -&gt; new primary database</td>
      <td><code>10.60.20.10</code> is bound to the new primary database.</td>
    </tr>
    <tr>
      <td>Application recovery</td>
      <td>Tier0 Enterprise replica on a Swarm node -&gt; <code>DB VIP</code> -&gt; new primary database</td>
      <td>The application connection pool reconnects, and read/write access is restored.</td>
    </tr>
  </tbody>
</table>

#### Key boundaries:

- `Witness` does not store business data, does not handle business read/write traffic, and does not expose database access to applications.
- Business services only connect to `DB VIP:5432`. Direct connections to `ent-db-01/02` are not allowed.
- `DB VIP` must be bound to the current PostgreSQL Primary node. It must not fail over based only on whether the database process is alive.
- `TimescaleDB` is deployed as a PostgreSQL extension on both database nodes. It is not separated into an independent database instance.
- The recommended design is `PostgreSQL streaming replication + repmgr + repmgr witness + keepalived DB VIP`.
- Automatic failover only handles primary database unavailability. After the original primary database recovers, it should rejoin as a standby database first. It should not automatically take the primary role back.

      <th>Failure Handling</th>

    <tr>

      <td><code>replicas: 1</code></td>
      <td>Stateless or weakly stateful</td>
      <td>If the current node fails, Swarm reschedules and restarts the service on another node.</td>
      <td>Data must be stored in PostgreSQL/TimescaleDB. Do not rely on local container files.</td>
    </tr>
    <tr>

      <td>Stateful</td>
      <td>If the fixed node fails, restore the node or recover from backup.</td>
      <td>Suitable for small-scale scenarios where a single point of failure in the messaging layer is acceptable.</td>
    </tr>
    <tr>
      <td>EMQX three-node cluster mode</td>
      <td>Run one fixed EMQX instance on each of the three Swarm nodes</td>
      <td>Stateful cluster</td>
      <td>If one EMQX node fails, the remaining EMQX nodes continue to serve traffic.</td>
      <td>Use this mode first if high availability is required for the messaging layer.</td>
    </tr>
    <tr>
      <td>APP VIP / HAProxy / Nginx</td>
      <td>Deployed on three nodes, with VIP failover</td>
      <td>Runtime configuration</td>
      <td>A healthy node takes over the VIP.</td>
      <td>Only used as an entry point. It does not store business data.</td>
    </tr>
    <tr>
      <td>NFS/NAS shared directory</td>
      <td>Mounted by all three Swarm nodes</td>
      <td>File state</td>
      <td>If NFS fails, functions that depend on shared files are affected.</td>
      <td>Only used for shared files such as uploads, imports, exports, and reports.</td>
    </tr>
  </tbody>
</table>

**Key rules:**

- Tier0 Enterprise can run as a single replica. Swarm only provides the cluster base and failover rescheduling capability.
- `EMQX` cannot use the same random single-replica rescheduling strategy as Tier0 Enterprise, because it has sessions, subscriptions, retained messages, rule engine state, and persistent directories.
- If `EMQX` is deployed as a single node, it must be treated as a single point of failure in the messaging layer and fixed to one Swarm node with a persistent directory.
- If the business cannot accept a single point of failure in `EMQX`, deploy a three-node EMQX cluster on the three Swarm nodes. Each node must have a fixed hostname, node name, and persistent directory.

### NFS/NAS Shared Storage Rules

NFS is introduced so that the Tier0 Enterprise single replica can still access the same business files after it is rescheduled between Swarm nodes. It does not change the boundaries of database primary-standby replication or the stateful EMQX service.

**Suitable content for NFS:**

- User-uploaded attachments
- Import/export files
- Report files
- Offline packages, templates, and reproducible intermediate files
- Business static files that must be read across nodes

**Content not recommended for NFS:**

- PostgreSQL / TimescaleDB data directories and WAL directories
- EMQX core data directories, Mnesia data, and runtime state
- Docker Engine data directory `/var/lib/docker`
- High-frequency database-like writes, lock-sensitive files, and strongly consistent queues

**NFS boundaries:**

- NFS is a shared file dependency. It is not a database high availability component.
- If NFS itself is a single point of failure, file-related functions will also have a single point of failure. In production, use an existing HA NAS or dual-controller storage.
- Before the Tier0 Enterprise single replica is rescheduled to another Swarm node, the target node must have NFS mounted and verified for read/write access.
- If Tier0 Enterprise does not write local files at all, NFS is not required.

## Service Deployment Recommendations

### Docker Swarm

- Set all three application nodes as `manager + worker`.
- Deploy Tier0 Enterprise by using `docker stack deploy`.
- Tier0 Enterprise uses `replicas: 1` by default. Swarm reschedules it to another healthy node after a node failure.
- Tier0 Enterprise containers do not store critical persistent data. Business data is stored in PostgreSQL/TimescaleDB.
- Each Swarm node has only `100G` storage and uses it as a single system disk. The Docker data directory stays on the system disk. No separate data disk is planned.
- Use `Docker config` for configuration files.
- Use `Docker secret` for passwords, database credentials, and certificate private keys.
- Use an on-site internal registry for container images to avoid public network dependencies during deployment.
- Configure health checks when publishing services to avoid forwarding traffic to containers that are not ready.
- If the application needs shared files across nodes, mount the shared directory from NFS/NAS. Do not rely on local temporary container directories for critical files.
- Configure cleanup policies for images, container logs, and build caches to avoid filling `/var/lib/docker`.

**Single-replica deployment example:**

```yaml
deploy:
  replicas: 1
  restart_policy:
    condition: any
  update_config:
    order: stop-first
```

### EMQX Stateful Service

`EMQX` must be handled as a stateful service. It must not be randomly rescheduled like a normal stateless Swarm replica.

  <thead>
    <tr>
      <th>Mode</th>
      <th>Recommended Scenario</th>
      <th>Deployment Rule</th>
      <th>Risk</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Single-node fixed mode</td>
      <td>Small-scale scenarios where short messaging-layer downtime is acceptable</td>
      <td>Fix it to <code>ent-swarm-01</code>. Use a local persistent directory and backup.</td>
      <td>If <code>ent-swarm-01</code> fails, EMQX cannot fail over automatically without data loss.</td>
    </tr>
    <tr>
      <td>Three-node cluster mode</td>
      <td>Production scenarios where a single point of failure in EMQX is not acceptable</td>
      <td>Run one EMQX node on each of the three Swarm nodes. Use fixed hostnames, node names, and data directories.</td>
      <td>Requires additional configuration for cluster ports, certificates, node discovery, and monitoring.</td>
    </tr>
  </tbody>
</table>

**Minimum requirements for single-node fixed mode:**

- Use a Swarm placement constraint to fix the node, for example `node.labels.emqx == true`.
- Fix the data directory to the host, for example `/data/emqx`.
- Include configurations, certificates, plugins, and rule engine configurations in backup.
- If the node fails, restore the original node first. If it cannot be restored, recover to a new node from backup. Do not handle it as a stateless random failover.

**Minimum requirements for three-node cluster mode:**

- Fix one EMQX instance on each Swarm node.
- Use a fixed node name for each EMQX node. The node name must not change when the container is recreated.
- Open cluster communication ports only between the three Swarm nodes.
- Expose the Dashboard only to the O&M network segment.
- MQTT access can be exposed through `APP VIP`, a separate MQTT VIP, or a layer 4 load balancer. Confirm the entry method before implementation.
- Do not place the EMQX core data directory on NFS. In single-node fixed mode, use the fixed node local disk and backup it properly. In three-node mode, rely on the EMQX cluster mechanism.

### NFS/NAS Shared Storage

- NFS/NAS is an optional external dependency and does not count as one of the six VMs.
- Mount the same shared directory on all three Swarm nodes.
- Use it only for shared files that Tier0 Enterprise needs to access across nodes.
- Do not use it for PostgreSQL/TimescaleDB data directories.
- Do not use it for EMQX core runtime data.
- Configure a pre-start mount check to avoid writing to a local directory after a service moves to a node where NFS is not mounted.
- If NFS fails, database read/write access is not directly affected, but functions such as uploads, imports, exports, and reports may fail.

### APP VIP

- Deploy `APP VIP` on the three Swarm nodes. Use `keepalived + HAProxy` or `keepalived + Nginx`.
- The VIP is only the unified entry. It does not require a separate VM.
- Health checks should use the application `/health` or `/ready` endpoint.
- If any Swarm node fails, the VIP moves to another healthy node.

### PostgreSQL + TimescaleDB

**Recommended baseline:**

- PostgreSQL: `16` or a version certified on site.
- TimescaleDB: choose a stable version that matches the PostgreSQL major version.
- Primary-standby replication: physical streaming replication.
- Failover: `repmgrd + witness`.
- Database entry: `DB VIP`.
- Connection pool: preferably use `PgBouncer` on the application side or in a separate container to avoid too many database connections.

### Witness

- The `Witness` node only stores `repmgr witness` metadata and helps database failure detection and arbitration.
- `Witness` does not store business data, does not handle business read/write traffic, and does not expose database access to applications.
- `Witness` needs stable network access. It does not require high storage performance.
- A `Witness` failure should not stop current business access, but it lowers the reliability of automatic failover.
- After `Witness` recovers, verify that it has rejoined the database cluster state.

## Backup, Monitoring, and Inspection

**Minimum delivery requirements:**

- Perform a full database backup every day and keep backups for at least `7` days.
- Enable WAL archiving or incremental backups to meet the on-site RPO.
- Perform a restore drill every month to verify that backups are usable.
- Monitor Swarm service health checks, container restart counts, node disk usage, memory, and CPU.
- Monitor EMQX connections, subscriptions, message throughput, queue backlog, rule engine failures, and persistent directory usage.
- Monitor NFS availability, mount status, capacity, inode usage, and read/write latency.
- Monitor PostgreSQL primary-standby delay, replication slots, WAL backlog, connections, slow SQL, and database bloat.
- Include the current VIP owner, repmgr cluster status, and Witness status in routine inspections.

**Recommended inspection commands:**

```bash
docker node ls
docker service ls
docker stack ps tier0-enterprise
ssh ent-db-01 'repmgr cluster show'
ssh ent-db-01 'psql -h 10.60.20.10 -p 5432 -c "select now();"'
ssh ent-db-01 'ip addr show | grep 10.60.20.10'
```

## Implementation Sequence

1. Create 6 VMs and fix hostnames, IP addresses, time synchronization, and SSH access.
2. Initialize system baselines, firewall rules, disk mounts, and kernel parameters.
3. If NFS/NAS is used, first complete shared directory setup, permissions, mounting, and restart auto-mount verification.
4. Deploy the three-node Docker Swarm.
5. Deploy `APP VIP + HAProxy/Nginx`.
6. Deploy EMQX in either single-node fixed mode or three-node cluster mode.
7. Deploy PostgreSQL + TimescaleDB primary-standby.
8. Deploy repmgr witness, repmgrd, and the DB VIP failover script.
9. Publish the Tier0 Enterprise single-replica stack.
10. Verify connectivity, NFS read/write access, EMQX recovery, database failover, backup restore, and routine inspection.
11. Finalize operation documents, archive account credentials, and set up routine inspection tasks.

## Risks and Items to Confirm

- The actual on-site network segment, gateway, DNS, NTP, certificates, and image registry address must be confirmed.
- It is not yet confirmed whether Tier0 Enterprise needs local file persistence. If it does, add a shared storage or object storage solution.
- It must be confirmed whether NFS/NAS is provided by existing on-site storage. If NFS is a single point of failure, shared file functions will also have a single point of failure.
- Each Swarm node has only a `100G` single system disk. Control the number of images, container logs, and temporary files. It is not suitable for long-term business data storage.
- `EMQX` is a stateful service. Before implementation, confirm whether to use single-node fixed mode or three-node cluster mode. In single-node fixed mode, the messaging layer has a single point of failure.
- Database RPO/RTO has not been confirmed. The current plan uses primary-standby automatic failover and cannot promise zero data loss.
- The `DB VIP` failover script must be verified on site, especially whether the virtualization platform allows VRRP.
- If the on-site network does not allow VRRP, use a load balancer, DNS switch, or virtualization-platform floating IP instead.
