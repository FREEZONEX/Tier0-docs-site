## 計画の目的

この計画は、オフラインまたはプライベート環境に 1 つの Tier0 Enterprise 環境をデプロイするためのものです。基本構成は 6 台の仮想マシンです。

- 3 台のアプリケーションノードで `Docker Swarm` クラスターを構成し、単一レプリカの Tier0 Enterprise サービスとステートフルな **EMQX** サービスを実行します。
- 2 台のデータベースノードで `PostgreSQL + TimescaleDB` の primary-standby クラスターを構成します。
- 1 台の `Witness` ノードが、データベース監視、仲裁、フェイルオーバーに参加します。

:::note[任意]
外部 `NFS/NAS` 共有ストレージを追加し、3 台の Swarm ノードからマウントできます。これは Tier0 Enterprise のノード間共有ファイルに使う外部依存であり、この計画の 6 台の業務 VM には含めません。
:::

:::tip[データベースリソースが変わる場合]
この計画では、各データベース VM が `4C / 8G / 500G` であることを前提にしています。データベースクラスター全体で `4C / 8G / 500G` が要件の場合は、実装前に性能と容量を再計算してください。
:::

## 全体トポロジー

<img class="t0-topology" src="/images/enterprise-ha-topology.png" alt="tier0-enterprise 6 VM 南北方向データ経路" />

図は主要なリンクのみを示しています。

- 業務アクセス経路: ユーザー / 現地システム -> `APP VIP` -> 3 台の `Docker Swarm` ノード -> `DB VIP` -> 現在の primary データベース。
- `APP VIP` は 3 台の Swarm ノードへトラフィックを分配します。Swarm ノードは `DB VIP` 経由でデータベースへアクセスします。
- `APP VIP` は 3 台の Swarm ノード間でフェイルオーバーできます。`DB VIP` は現在の PostgreSQL Primary ロールに追従し、2 台のデータベースノード間で切り替わります。
- `Witness` はデータベース仲裁、primary 選択、ヘルスチェックのみに参加し、業務アクセス経路には入りません。
- `EMQX` はステートフルサービスです。デフォルトでは永続ストレージを持つ 1 台のノードに固定します。メッセージング層にも HA が必要な場合は、3 ノード EMQX クラスターをデプロイしてください。

## データベース監視とフェイルオーバー

| フェーズ | リンク | 説明 |
| --- | --- | --- |
| 通常アクセス | Swarm ノード上の Tier0 Enterprise レプリカ -> `DB VIP` -> 現在の primary | アプリケーションは常に `10.60.20.10:5432` のみに接続します。 |
| 通常レプリケーション | 現在の primary -> 現在の standby | PostgreSQL WAL streaming replication でデータを同期します。 |
| 通常監視 | `DB-01` / `DB-02` -> `Witness` | `repmgrd` が状態を報告し、`Witness` が仲裁に参加します。 |
| primary 障害 | standby -> `Witness` | standby が split-brain を避けるために仲裁を要求します。 |
| standby 昇格 | standby `promote` | `Witness` 確認後、standby が新しい primary に昇格します。 |
| VIP フェイルオーバー | `DB VIP` -> 新 primary | `10.60.20.10` を新しい primary に紐づけます。 |
| アプリケーション復旧 | Tier0 Enterprise -> `DB VIP` -> 新 primary | 接続プールが再接続し、読み書きが復旧します。 |

重要な境界:

- 業務サービスは `DB VIP:5432` のみに接続し、`ent-db-01/02` への直接接続は許可しません。
- `Witness` は業務データを保存せず、業務の読み書きを処理せず、アプリケーションへデータベースアクセスを公開しません。
- `TimescaleDB` は両方のデータベースノードに PostgreSQL extension としてデプロイします。
- 推奨構成は `PostgreSQL streaming replication + repmgr + repmgr witness + keepalived DB VIP` です。
- 自動フェイルオーバーは primary の利用不可のみを扱います。元 primary が復旧した後は、まず standby として再参加させます。

## デプロイ推奨事項

- `docker stack deploy` で Tier0 Enterprise をデプロイします。
- 設定ファイルには `Docker config`、パスワード、データベース認証情報、証明書秘密鍵には `Docker secret` を使います。
- デプロイ時に公開ネットワークへ依存しないよう、現地の内部イメージレジストリを使います。
- `APP VIP` は 3 台の Swarm ノード上に `keepalived + HAProxy` または `keepalived + Nginx` でデプロイします。
- データベース入口は `DB VIP` に統一します。接続数が増えすぎないよう、アプリケーション側で接続プールを使うことを推奨します。
- NFS/NAS は Tier0 Enterprise のノード間共有ファイル専用にし、PostgreSQL/TimescaleDB のデータディレクトリや EMQX の中核 runtime データには使わないでください。

## ポートとファイアウォール

完全なポート一覧は [標準ポート一覧](../standard-port-list/) を参照してください。

デフォルトでは開けないポート:

- Docker remote API `2375` / `2376`。使う場合も O&M ネットワークに限定してください。
- データベースノードの `80/443`。データベースノードは Web 入口を提供しません。
- アプリケーションから Witness への `5432`。Witness は業務 SQL を処理しません。

## 高可用機能と障害境界

| 障害シナリオ | システムの動作 | 境界 |
| --- | --- | --- |
| 現在の Swarm ノードが故障 | Swarm が単一レプリカサービスを別の正常ノードへ再スケジュールします。 | コンテナ再起動とアプリケーション再接続を待つ必要があります。 |
| 単一ノード固定モードの EMQX が故障 | まず元ノードを復旧します。復旧できない場合はバックアップから新ノードへ復元します。 | メッセージング層は中断し、stateless サービスのような無損失切替はできません。 |
| 3 ノード EMQX クラスターの 1 ノードが故障 | 残りのノードが継続してサービスし、復旧後に故障ノードが再参加します。 | EMQX クラスター構成を事前に完了しておく必要があります。 |
| 現在の primary が故障 | standby が Witness 仲裁後に昇格し、`DB VIP` が切り替わります。 | 書き込みが数秒から数分中断します。アプリケーションは再接続が必要です。 |
| standby が故障 | primary は継続してサービスし、レプリケーション中断アラートが出ます。 | 自動フェイルオーバー能力が低下します。 |
| Witness が故障 | 現在の primary は継続してサービスします。 | 自動フェイルオーバー能力が低下するため、Witness を優先的に復旧します。 |
| NFS/NAS が故障 | 共有ファイルに依存する機能が影響を受けます。 | データベースやメモリ上で動作中の業務プロセスは必ずしも中断しません。 |

## バックアップ、監視、点検

- データベースを毎日バックアップし、大きな変更前には必ずバックアップします。
- APP VIP、DB VIP、PostgreSQL replication lag、Swarm ノード状態、EMQX 状態、ディスク使用率を監視します。
- primary 切替、アプリケーションノード障害、バックアップ復元、共有ストレージ障害を定期的に演習します。
- 本番前に、SSH パスワードレス接続、NTP 時刻同期、イメージレジストリ、証明書、DNS、gateway、ファイアウォールルールを確認します。

## 実装順序

1. 現地ネットワーク、VIP、DNS、NTP、証明書、イメージレジストリ、ファイアウォールポリシーを確認します。
2. 6 台の VM の OS、ディスク、時刻同期、基本セキュリティ設定を準備します。
3. NFS/NAS を使う場合は、共有ディレクトリ、権限、マウント、再起動時の自動マウントを検証します。
4. 3 ノード Docker Swarm をデプロイします。
5. `APP VIP + HAProxy/Nginx` をデプロイします。
6. 必要に応じて単一ノード固定 EMQX または 3 ノード EMQX クラスターをデプロイします。
7. PostgreSQL + TimescaleDB primary-standby をデプロイします。
8. repmgr witness、repmgrd、DB VIP フェイルオーバースクリプトをデプロイします。
9. Tier0 Enterprise 単一レプリカ stack を公開します。

## リスクと確認項目

- 現地ネットワークセグメント、gateway、DNS、NTP、証明書、イメージレジストリのアドレスを確認してください。
- Tier0 Enterprise がローカルファイル永続化を必要とするか確認してください。必要な場合は共有ストレージまたはオブジェクトストレージを追加します。
- `DB VIP` フェイルオーバースクリプトは現地で検証してください。特に仮想化基盤が VRRP を許可するか確認が必要です。
- 現地ネットワークで VRRP が許可されない場合は、ロードバランサー、DNS 切替、または仮想化基盤の floating IP を使ってください。
