## Demo UNS の構造
:::tip[What needs to be done first]
Before modeling data, answer the following questions.
- UNS からどのデータを利用するか？
- UNS にどのデータを戻すか？
:::


### 物理データ

物理的な場所を持つ機械や設備の産業データは、通常 ISA-95 標準に基づいてモデル化します。

#### ISA-95 設備階層
ISA-95 の設備階層は、組織内の物理資産と制御システムがどのように構造化されるかを示します。
It is broken into the following levels:
- **Enterprise**：最上位レベルで、組織全体を表します。全体の業務プロセスや意思決定エンティティを含み、通常は ERP システムに支えられます。  
- **Site**：特定の地理的位置や施設を表します。各 site は独自の管理システムとローカル生産制御を持つことができます。 
- **Area**：site 内の運用区分で、包装、組立、倉庫などの機能エリアであることが多いです。 
- **Process Cell**：area 内の製造の基本単位です。特定の製品を生産したり特定の機能を実行したりするために必要な設備を含みます。 
- **Unit**：process cell 内の明確な部分で、混合、加熱、組立など特定の機能を担います。 
- **Equipment Module**：unit 内で特定のタスクを実行するモジュール化された設備です。たとえば混合 unit では、加熱要素がこれに当たります。  
- **Control Module**：最下位レベルで、センサー、ポンプ、アクチュエーター、PID 制御ループなど、物理プロセスを直接制御する個別のデバイスやコントローラーを表します。

### 業務データ

ERP の生産注文や作業指示計画など、業務計画やプロセスオブジェクトについては、業務コンテキストに沿ってデータをモデル化します。
:::tip[How to recognize business data]
Data that does not have a physical location.
:::
#### 業務データ Modeling Guide
```
Factory / Business Domain / Business Object / Topic Type / Topic
```
- **Business Domain**：データが属する業務プロセスまたはシステムを確認します。
    ```
    ERP
    MES
    WMS
    QMS
    EAM
    LIMS
    Planning
    Inventory
    Maintenance
    Quality
    Logistics
    ...
    ```
- **Business Object**：業務ドメインで管理される中心オブジェクトです。
    ```
    ProductionOrders
    WorkOrderPlan
    MaterialInventory
    QualityInspection
    MaintenanceOrders
    PurchaseOrders
    Shipments
    ...
    ```
- **Topic Type**：**Metric**、**Action**、**State** からデータタイプを選択します。

    | Topic Type | Used for | Examples |
    | --- | --- | --- |
    | STATE | 現在の状態または現在の事実 | 作業指示の状態、注文状態、設備状態 |
    | METRIC | 継続的に変化する数値 | 温度、速度、生産量、OEE、エネルギー消費 |
    | ACTION | 外部実行者へのコマンド | 設備起動、設備停止、アラームリセット、作業指示の払い出し要求 |

## Topic Explosion を避ける方法
Tier0 uses 3 fixed topic types: `Metric`, `Action`, and `State` to avoid topic explosion. 

- **高頻度リアルタイムデータには `Metric` を使う**

  高頻度の測定値やテレメトリは、各信号の詳細ごとに namespace を広げるのではなく `Metric` に配置します。

  <Tabs syncKey="topic-explosion-metric">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Machine_001
          ├── Temperature_Value
          ├── Temperature_Unit
          ├── Temperature_Status
          └── Speed_Value
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Metric
          └── Machine_001
  ```

  Model fields:

  ```json
  {
    "temperature": 80,
    "temperature_unit": "Celsius",
    "temperature_status": "Normal",
    "speed": 300
  }
  ```

    </TabItem>
  </Tabs>

- **低頻度ステータスデータには `State` を使う**

  低頻度の状態、スナップショット、現在の業務状態は `State` に配置します。ジョブ、バッチ、注文などの変化するエンティティごとに新しい topic を作らないようにします。

  <Tabs syncKey="topic-explosion-state">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── State
          ├── Job_001
          ├── Job_002
          └── Job_003
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── State
          └── Machine_001
  ```

  Model fields:

  ```json
  {
    "current_job": "Job_003",
    "product": "Product_A"
  }
  ```

    </TabItem>
  </Tabs>

- **低頻度アクションデータには `Action` を使う**

  コマンド、リクエスト、オペレーター操作は `Action` に配置します。action の入力はパラメータごとに topic を作るのではなく、topic のフィールドとして保持します。

  <Tabs syncKey="topic-explosion-action">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Machine_001
          ├── Start_Command
          ├── Target_Height
          └── Light_On
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Action
          └── StartJob
  ```

  Model fields:

  ```json
  {
    "target_height": 12.5,
    "light_on": true,
    "requested_by": "operator"
  }
  ```

    </TabItem>
  </Tabs>

## Payload 構造

topic payload を設計するときは、消費されるデータと書き戻されるデータの両方を考慮します。

- **Action**

  Action payload には、データが Tier0 データベースに保存された時刻を記録する `_timestamp` が自動的に含まれます。

  <Tabs syncKey="payload-action">
    <TabItem label="start_job">

  ```json
  {
    "light_turn_on": true,
    "height_adjust": 12.5,
    "_timestamp": "2026-07-10T10:00:00Z"
  }
  ```

    </TabItem>
    <TabItem label="stop_job">

  ```json
  {
    "water_pump_off": true,
    "window_close": true,
    "_timestamp": "2026-07-10T10:00:00Z"
  }
  ```

    </TabItem>
  </Tabs>

- **Metric**

  Metric payload には、Tier0 データベースへの保存時刻を記録する `_timestamp` と、データ品質を表す `_quality` が自動的に含まれます。

  ```json
    "plan_number": "WOP-20260710-001",

    "updated_at": "2026-07-10T10:00:00Z",
    "_timestamp": "2026-07-10T10:00:00Z",
    "_quality": "Good"
  }
  ```

    </TabItem>
    <TabItem label="Site_01 / Target_Qty">

  ```json
  {
    "production_order_id": "PO-20260710-001",
    "plan_id": "PLAN-001",
    "plan_number": "WOP-20260710-001",

    "value": 1200,
    "unit": "pcs",

    "_timestamp": "2026-07-10T10:00:00Z",
    "_quality": "Good"
  }
  ```

    </TabItem>
  </Tabs>

  State payload には、データが Tier0 データベースに保存された時刻を記録する `_timestamp` が自動的に含まれます。
