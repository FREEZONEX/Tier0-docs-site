### Datos físicos

Los datos industriales de máquinas y equipos con ubicaciones físicas se modelan normalmente según el estándar ISA-95.

#### Jerarquía de equipos ISA-95
La jerarquía de equipos ISA-95 describe cómo se estructuran los activos físicos y los sistemas de control dentro de una organización.
It is broken into the following levels:
- **Enterprise**: El nivel superior, que representa toda la organización. Incluye procesos de negocio generales y entidades de decisión, normalmente soportadas por sistemas ERP.  
- **Site**: Representa una ubicación geográfica o instalación específica. Cada site puede tener sus propios sistemas de gestión y control local de producción. 
- **Area**: Una división operativa dentro de un site, a menudo un área funcional como empaque, ensamblaje o almacén. 
- **Process Cell**: La unidad básica de fabricación dentro de un area. Incluye todos los equipos necesarios para producir un producto específico o realizar una función específica. 
- **Unit**: Una parte diferenciada del process cell, responsable de una función específica como mezclar, calentar o ensamblar. 
- **Equipment Module**: Un equipo modular que realiza una tarea específica dentro de un unit. Por ejemplo, en un unit de mezcla, el módulo podría ser un elemento calefactor.  
- **Control Module**: El nivel más bajo, que representa dispositivos o controladores individuales como sensores, bombas, actuadores o lazos PID que controlan directamente procesos físicos.

### Datos de negocio

Para objetos de planificación y proceso de negocio, como órdenes de producción y planes de trabajo en un ERP, modela los datos según su contexto de negocio.
:::tip[How to recognize business data]
Data that does not have a physical location.
:::
#### Guía de modelado de datos de negocio
```
Factory / Business Domain / Business Object / Topic Type / Topic
```
- **Business Domain**: Confirma el proceso o sistema de negocio al que pertenecen los datos.
    ```
    ERP
    MES
    WMS
    QMS
    EAM
    LIMS
    Planning
    Inventory
    Maintenance
    Quality
    Logistics
    ...
    ```
- **Business Object**: El objeto gestionado principal dentro del dominio de negocio.
    ```
    ProductionOrders
    WorkOrderPlan
    MaterialInventory
    QualityInspection
    MaintenanceOrders
    PurchaseOrders
    Shipments
    ...
    ```
- **Topic Type**: Elige el tipo de dato entre **Metric**, **Action** y **State**.

    | Topic Type | Used for | Examples |
    | --- | --- | --- |
    | STATE | Estado actual o hechos actuales | Estado de orden de trabajo, estado de orden, estado de equipo |
    | METRIC | Valores numéricos que cambian continuamente | Temperatura, velocidad, producción, OEE, consumo de energía |
    | ACTION | Comandos para ejecutores externos | Iniciar equipo, detener equipo, restablecer alarma, solicitar despacho de orden de trabajo |

## Cómo evitar la explosión de topics
Tier0 uses 3 fixed topic types: `Metric`, `Action`, and `State` to avoid topic explosion. 

- **Usa `Metric` para datos en tiempo real de alta frecuencia**

  Coloca mediciones y telemetría de alta frecuencia bajo `Metric` en lugar de ampliar el namespace para cada detalle de señal.

  <Tabs syncKey="topic-explosion-metric">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Machine_001
          ├── Temperature_Value
          ├── Temperature_Unit
          ├── Temperature_Status
          └── Speed_Value
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Metric
          └── Machine_001
  ```

  Model fields:

  ```json
  {
    "temperature": 80,
    "temperature_unit": "Celsius",
    "temperature_status": "Normal",
    "speed": 300
  }
  ```

    </TabItem>
  </Tabs>

- **Usa `State` para datos de estado de baja frecuencia**

  Coloca estados de baja frecuencia, snapshots y estado de negocio actual bajo `State`. Evita crear un topic nuevo para cada entidad cambiante, como trabajos, lotes u órdenes.

  <Tabs syncKey="topic-explosion-state">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── State
          ├── Job_001
          ├── Job_002
          └── Job_003
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── State
          └── Machine_001
  ```

  Model fields:

  ```json
  {
    "current_job": "Job_003",
    "product": "Product_A"
  }
  ```

    </TabItem>
  </Tabs>

- **Usa `Action` para datos de acción de baja frecuencia**

  Coloca comandos, solicitudes y acciones de operador bajo `Action`. Mantén las entradas de la acción como campos de un topic en lugar de crear un topic por parámetro.

  <Tabs syncKey="topic-explosion-action">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Machine_001
          ├── Start_Command
          ├── Target_Height
          └── Light_On
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Action
          └── StartJob
  ```

  Model fields:

  ```json
  {
    "target_height": 12.5,
    "light_on": true,
    "requested_by": "operator"
  }
  ```

    </TabItem>
  </Tabs>

## Estructura del payload

Al estructurar el topic payload, considera tanto los datos que se consumen como los datos que se escriben de vuelta.

- **Action**

  Los payloads de Action incluyen automáticamente `_timestamp`, que registra cuándo se almacenan los datos en la base de datos de Tier0.

  ```json
    "height_adjust": 12.5,
    "_timestamp": "2026-07-10T10:00:00Z"
  }
  ```

    </TabItem>
