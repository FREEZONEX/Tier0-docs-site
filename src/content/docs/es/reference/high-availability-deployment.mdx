## Objetivo del plan

Este plan despliega un entorno Tier0 Enterprise en una instalación offline o privada. La base usa 6 máquinas virtuales:

- 3 nodos de aplicación forman el clúster `Docker Swarm` y ejecutan el servicio Tier0 Enterprise de réplica única y el servicio **EMQX** con estado.
- 2 nodos de base de datos forman el clúster primario-en-espera de `PostgreSQL + TimescaleDB`.
- 1 nodo `Witness` participa en la supervisión, arbitraje y failover de la base de datos.

:::note[Opcional]
Se puede agregar almacenamiento compartido externo `NFS/NAS` y montarlo en los tres nodos Swarm. Se usa para archivos compartidos de Tier0 Enterprise entre nodos, es una dependencia externa y no cuenta como una de las 6 VM de negocio.
:::

:::tip[Cuando cambien los recursos de base de datos]
Este plan asume que cada VM de base de datos tiene `4C / 8G / 500G`. Si el requisito es `4C / 8G / 500G` para todo el clúster de base de datos, la capacidad y el rendimiento deben recalcularse antes de la implementación.
:::

## Topología general

<img class="t0-topology" src="/images/enterprise-ha-topology.png" alt="tier0-enterprise ruta de datos norte-sur de seis VM" />

El diagrama muestra solo los enlaces principales:

- Ruta de acceso de negocio: usuarios / sistemas del sitio -> `APP VIP` -> tres nodos `Docker Swarm` -> `DB VIP` -> base de datos primaria actual.
- `APP VIP` distribuye tráfico a los tres nodos Swarm. Los nodos Swarm acceden a la base de datos mediante `DB VIP`.
- `APP VIP` puede hacer failover entre los tres nodos Swarm. `DB VIP` sigue el rol PostgreSQL Primary actual y puede hacer failover entre los dos nodos de base de datos.
- `Witness` solo participa en arbitraje, selección de primario y health checks. No está en la ruta de negocio.
- `EMQX` es un servicio con estado. Por defecto debe fijarse a un nodo con almacenamiento persistente; si la capa de mensajería requiere HA, despliega un clúster EMQX de tres nodos.

## Supervisión y failover de base de datos

| Fase | Enlace | Descripción |
| --- | --- | --- |
| Acceso normal | Réplica Tier0 Enterprise en un nodo Swarm -> `DB VIP` -> primario actual | La aplicación siempre conecta solo a `10.60.20.10:5432`. |
| Replicación normal | Primario actual -> standby actual | Los datos se sincronizan con PostgreSQL WAL streaming replication. |
| Supervisión normal | `DB-01` / `DB-02` -> `Witness` | `repmgrd` reporta estado y `Witness` participa en el arbitraje. |
| Fallo del primario | Standby -> `Witness` | El standby solicita arbitraje para evitar split-brain. |
| Promoción del standby | Standby `promote` | Tras la confirmación de `Witness`, el standby se convierte en el nuevo primario. |
| Failover de VIP | `DB VIP` -> nuevo primario | `10.60.20.10` se vincula al nuevo primario. |
| Recuperación de aplicación | Tier0 Enterprise -> `DB VIP` -> nuevo primario | El pool de conexiones reconecta y se restaura la lectura/escritura. |

Límites clave:

- Los servicios de negocio solo conectan a `DB VIP:5432`; no se permiten conexiones directas a `ent-db-01/02`.
- `Witness` no almacena datos de negocio, no procesa lectura/escritura de negocio y no expone acceso de base de datos a aplicaciones.
- `TimescaleDB` se despliega como extensión de PostgreSQL en ambos nodos de base de datos.
- El diseño recomendado es `PostgreSQL streaming replication + repmgr + repmgr witness + keepalived DB VIP`.
- El failover automático solo cubre indisponibilidad del primario. Cuando el primario original se recupere, debe volver primero como standby.

## Recomendaciones de despliegue

- Despliega Tier0 Enterprise con `docker stack deploy`.
- Usa `Docker config` para archivos de configuración y `Docker secret` para contraseñas, credenciales de base de datos y claves privadas de certificados.
- Usa un registro interno de imágenes para evitar dependencias de red pública durante el despliegue.
- Despliega `APP VIP` en los tres nodos Swarm con `keepalived + HAProxy` o `keepalived + Nginx`.
- Usa `DB VIP` como entrada única de base de datos. Se recomienda un pool de conexiones para evitar demasiadas conexiones a la base de datos.
- Usa NFS/NAS solo para archivos compartidos de Tier0 Enterprise entre nodos; no lo uses para directorios de datos de PostgreSQL/TimescaleDB ni para datos runtime centrales de EMQX.

## Puertos y firewall

Consulta la [lista estándar de puertos](../standard-port-list/) para el detalle completo.

No abras por defecto:

- API remota de Docker `2375` / `2376`, salvo que esté controlada y limitada a la red de O&M.
- `80/443` en nodos de base de datos; esos nodos no alojan entradas Web.
- Acceso de aplicaciones a `5432` en Witness; Witness no procesa SQL de negocio.

## Capacidades HA y límites de fallo

| Escenario | Comportamiento del sistema | Límite |
| --- | --- | --- |
| Falla el nodo Swarm actual | Swarm reprograma el servicio de réplica única en otro nodo sano. | Hay que esperar reinicio de contenedor y reconexión de la aplicación. |
| Falla EMQX en modo fijo de un nodo | Recupera primero el nodo original; si no es posible, restaura desde backup en un nodo nuevo. | La capa de mensajería se interrumpe y no hace failover sin pérdida como un servicio stateless. |
| Falla un nodo en clúster EMQX de tres nodos | Los nodos restantes siguen atendiendo tráfico y el nodo fallido se reincorpora tras recuperarse. | Requiere configuración previa del clúster EMQX. |
| Falla el primario actual | El standby se promueve tras arbitraje de Witness y `DB VIP` hace failover. | Interrupción de escritura de segundos a minutos; la aplicación debe reconectar. |
| Falla el standby | El primario sigue atendiendo y se genera alerta de replicación. | Se reduce la capacidad de failover automático. |
| Falla Witness | El primario actual sigue atendiendo. | Se reduce la capacidad de failover automático; recupera Witness primero. |
| Falla NFS/NAS | Las funciones que dependen de archivos compartidos se ven afectadas. | La base de datos y procesos ya en memoria pueden no interrumpirse. |

## Backup, monitoreo e inspección

- Ejecuta backups diarios de base de datos y realiza backup obligatorio antes de cambios mayores.
- Monitorea APP VIP, DB VIP, PostgreSQL replication lag, estado de nodos Swarm, estado de EMQX y uso de disco.
- Ensaya periódicamente failover de primario, fallo de nodo de aplicación, restauración de backup y fallo de almacenamiento compartido.
- Antes de salir a producción, confirma SSH sin contraseña, sincronización NTP, registro de imágenes, certificados, DNS, gateway y reglas de firewall.

## Secuencia de implementación

1. Confirma red del sitio, VIP, DNS, NTP, certificados, registro de imágenes y políticas de firewall.
2. Prepara el sistema operativo, discos, sincronización horaria y configuración básica de seguridad en las 6 VM.
3. Si se usa NFS/NAS, valida directorios compartidos, permisos, montaje y montaje automático al reiniciar.
4. Despliega el Docker Swarm de tres nodos.
5. Despliega `APP VIP + HAProxy/Nginx`.
6. Despliega EMQX en modo fijo de un nodo o como clúster de tres nodos.
7. Despliega PostgreSQL + TimescaleDB primario-standby.
8. Despliega repmgr witness, repmgrd y el script de failover de DB VIP.
9. Publica el stack Tier0 Enterprise de réplica única.

## Riesgos y elementos por confirmar

- Confirma segmento de red, gateway, DNS, NTP, certificados y dirección del registro de imágenes.
- Confirma si Tier0 Enterprise requiere persistencia local de archivos; si la requiere, agrega almacenamiento compartido u objeto.
- El script de failover de `DB VIP` debe verificarse en sitio, especialmente si la plataforma de virtualización permite VRRP.
- Si la red del sitio no permite VRRP, usa balanceador, cambio DNS o IP flotante de la plataforma de virtualización.
