## Plan Objective

Este plan despliega un entorno Tier0 Enterprise en un entorno offline o privado. Usa 6 máquinas virtuales en total:

- 3 application nodes: form the `Docker Swarm` cluster base and run the single-replica Tier0 Enterprise service and the stateful **EMQX** service.
- 2 database nodes: form the `PostgreSQL + TimescaleDB` primary-standby cluster.
- 1 `Witness` node: participates in database monitoring, arbitration, and failover.

:::note[Optional]
- Se puede agregar almacenamiento compartido externo `NFS/NAS` y montarlo en los tres nodos Swarm. Se usa para archivos compartidos de Tier0 Enterprise entre nodos.
- This shared storage is an external dependency and is not counted as one of the 6 business VMs in this plan.
:::

:::tip[When database resource changes]
Este plan asume que cada VM de base de datos tiene `4C / 8G / 500G`. Si el requisito del usuario es `4C / 8G / 500G` para todo el clúster de base de datos, el rendimiento y la capacidad deben recalcularse antes de la implementación.
:::

## Overall Topology

<img width={650} src="/images/enterprise-ha-topology.png" />

:::note
The diagram only shows the core links.
:::

- Business access path: users / on-site systems -> `APP VIP` -> three `Docker Swarm` nodes -> `DB VIP` -> current primary database.
- La topología está organizada de arriba abajo para mostrar entrada de solicitudes, procesamiento de aplicación, acceso a base de datos y monitoreo de alta disponibilidad.
- El diagrama de topología no muestra puertos internos del clúster, enlaces SSH de O&M ni enlaces de monitoreo. Estos se describen en la <a href="#connection-list">Lista de conexiones</a> y la <a href="#detailed-port-list">Lista detallada de puertos</a>.
- `Witness` solo participa en arbitraje de base de datos, selección de primary y health checks. No está en la ruta de acceso de negocio.
- `APP VIP` distributes traffic to the three Swarm nodes. The Swarm nodes access the database through `DB VIP`.
- `APP VIP` puede hacer failover entre los tres nodos Swarm. `DB VIP` sigue el rol PostgreSQL Primary actual y puede hacer failover entre los dos nodos de base de datos.
- Tier0 Enterprise se ejecuta como servicio de réplica única. Si falla el nodo actual, Swarm lo reprograma y reinicia en otro nodo saludable.
- `EMQX` es un servicio con estado. No debe reprogramarse aleatoriamente como un servicio stateless normal. Por defecto debe fijarse a un nodo con almacenamiento persistente. Si la capa de mensajería requiere alta disponibilidad, despliega un clúster EMQX de tres nodos.
- El `NFS/NAS` opcional solo se usa como almacenamiento de archivos compartidos para los nodos Swarm. No forma parte de la ruta principal de solicitudes de base de datos.

## Database Monitoring and Failover

#### Database failover path:

<table>
  <thead>
    <tr>
      <th>Stage</th>
      <th>Link</th>
      <th>Description</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Normal access</td>
      <td>Tier0 Enterprise replica on a Swarm node -&gt; <code>DB VIP</code> -&gt; current primary database</td>
      <td>The application always connects only to <code>10.60.20.10:5432</code>.</td>
    </tr>
    <tr>
      <td>Normal replication</td>
      <td>Current primary database -&gt; current standby database</td>
      <td>Data is synchronized through PostgreSQL WAL streaming replication.</td>
    </tr>
    <tr>
      <td>Normal monitoring</td>
      <td><code>DB-01</code> / <code>DB-02</code> -&gt; <code>Witness</code></td>
      <td><code>repmgrd</code> reports status, and <code>Witness</code> participates in arbitration.</td>
    </tr>
    <tr>
      <td>Primary database failure</td>
      <td>Standby database -&gt; <code>Witness</code></td>
      <td>The standby database requests arbitration to avoid split-brain.</td>
    </tr>
    <tr>
      <td>Standby promotion</td>
      <td>Standby database <code>promote</code></td>
      <td>After <code>Witness</code> confirms the failover, the standby database is promoted to the new primary database.</td>
    </tr>
    <tr>
      <td>VIP failover</td>
      <td><code>DB VIP</code> -&gt; new primary database</td>
      <td><code>10.60.20.10</code> is bound to the new primary database.</td>
    </tr>
    <tr>
      <td>Application recovery</td>
      <td>Tier0 Enterprise replica on a Swarm node -&gt; <code>DB VIP</code> -&gt; new primary database</td>
      <td>The application connection pool reconnects, and read/write access is restored.</td>
    </tr>
  </tbody>
</table>

#### Key boundaries:

- `Witness` no almacena datos de negocio, no maneja tráfico de lectura/escritura de negocio y no expone acceso a base de datos para las aplicaciones.
- Business services only connect to `DB VIP:5432`. Direct connections to `ent-db-01/02` are not allowed.
- `DB VIP` debe estar vinculado al nodo PostgreSQL Primary actual. No debe hacer failover basándose solo en si el proceso de base de datos está vivo.
- `TimescaleDB` is deployed as a PostgreSQL extension on both database nodes. It is not separated into an independent database instance.
- The recommended design is `PostgreSQL streaming replication + repmgr + repmgr witness + keepalived DB VIP`.
- El failover automático solo maneja la indisponibilidad de la base de datos primary. Después de que el primary original se recupere, primero debe reincorporarse como standby. No debe recuperar automáticamente el rol primary.

      <td>Use this mode first if high availability is required for the messaging layer.</td>

      <td>Deployed on three nodes, with VIP failover</td>

      <td>Only used as an entry point. It does not store business data.</td>
    </tr>
    <tr>
      <td>NFS/NAS shared directory</td>
      <td>Mounted by all three Swarm nodes</td>
      <td>File state</td>
      <td>If NFS fails, functions that depend on shared files are affected.</td>
      <td>Only used for shared files such as uploads, imports, exports, and reports.</td>
    </tr>
  </tbody>

**Key rules:**

- Tier0 Enterprise can run as a single replica. Swarm only provides the cluster base and failover rescheduling capability.
- `EMQX` no puede usar la misma estrategia de reprogramación aleatoria de réplica única que Tier0 Enterprise, porque tiene sesiones, suscripciones, mensajes retenidos, estado del motor de reglas y directorios persistentes.
- Si `EMQX` se despliega como un solo nodo, debe tratarse como punto único de fallo en la capa de mensajería y fijarse a un nodo Swarm con directorio persistente.
- Si el negocio no puede aceptar un punto único de fallo en `EMQX`, despliega un clúster EMQX de tres nodos sobre los tres nodos Swarm. Cada nodo debe tener hostname, node name y directorio persistente fijos.

### NFS/NAS Shared Storage Rules

NFS se introduce para que la réplica única de Tier0 Enterprise pueda seguir accediendo a los mismos archivos de negocio después de ser reprogramada entre nodos Swarm. No cambia los límites de la replicación primaria-standby de la base de datos ni del servicio EMQX con estado.

**Suitable content for NFS:**

- User-uploaded attachments
- Import/export files
- Report files
- Offline packages, templates, and reproducible intermediate files
- Business static files that must be read across nodes

**Content not recommended for NFS:**

- PostgreSQL / TimescaleDB data directories and WAL directories
- EMQX core data directories, Mnesia data, and runtime state
- Docker Engine data directory `/var/lib/docker`
- High-frequency database-like writes, lock-sensitive files, and strongly consistent queues

**NFS boundaries:**

- NFS is a shared file dependency. It is not a database high availability component.
- Si NFS en sí es un punto único de fallo, las funciones relacionadas con archivos también tendrán un punto único de fallo. En producción, usa un NAS HA existente o almacenamiento de doble controlador.
- Antes de reprogramar la réplica única de Tier0 Enterprise a otro nodo Swarm, el nodo destino debe tener NFS montado y verificado con acceso de lectura/escritura.
- If Tier0 Enterprise does not write local files at all, NFS is not required.

## Service Deployment Recommendations

### Docker Swarm

- Set all three application nodes as `manager + worker`.
- Deploy Tier0 Enterprise by using `docker stack deploy`.
- Tier0 Enterprise uses `replicas: 1` by default. Swarm reschedules it to another healthy node after a node failure.
- Tier0 Enterprise containers do not store critical persistent data. Business data is stored in PostgreSQL/TimescaleDB.
- Cada nodo Swarm tiene solo `100G` de almacenamiento y lo usa como un único disco de sistema. El directorio de datos de Docker permanece en el disco de sistema. No se planifica un disco de datos separado.
- Use `Docker config` for configuration files.
- Use `Docker secret` for passwords, database credentials, and certificate private keys.
- Use an on-site internal registry for container images to avoid public network dependencies during deployment.
- Configura health checks al publicar servicios para evitar reenviar tráfico a contenedores que no están listos.
- Si la aplicación necesita archivos compartidos entre nodos, monta el directorio compartido desde NFS/NAS. No dependas de directorios temporales locales del contenedor para archivos críticos.
- Configure cleanup policies for images, container logs, and build caches to avoid filling `/var/lib/docker`.

**Single-replica deployment example:**

```yaml
deploy:
  replicas: 1
  restart_policy:
    condition: any
  update_config:
    order: stop-first
```

### EMQX Stateful Service

`EMQX` must be handled as a stateful service. It must not be randomly rescheduled like a normal stateless Swarm replica.

<table>
  <thead>
    <tr>
      <th>Mode</th>
      <th>Recommended Scenario</th>
      <th>Deployment Rule</th>
      <th>Risk</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Single-node fixed mode</td>
      <td>Small-scale scenarios where short messaging-layer downtime is acceptable</td>
      <td>Fix it to <code>ent-swarm-01</code>. Use a local persistent directory and backup.</td>
      <td>If <code>ent-swarm-01</code> fails, EMQX cannot fail over automatically without data loss.</td>
    </tr>
    <tr>
      <td>Three-node cluster mode</td>
      <td>Production scenarios where a single point of failure in EMQX is not acceptable</td>
      <td>Run one EMQX node on each of the three Swarm nodes. Use fixed hostnames, node names, and data directories.</td>
      <td>Requires additional configuration for cluster ports, certificates, node discovery, and monitoring.</td>
    </tr>
  </tbody>
</table>

**Minimum requirements for single-node fixed mode:**

- Use a Swarm placement constraint to fix the node, for example `node.labels.emqx == true`.
- Fix the data directory to the host, for example `/data/emqx`.
- Include configurations, certificates, plugins, and rule engine configurations in backup.
- Si el nodo falla, restaura primero el nodo original. Si no puede restaurarse, recupera en un nodo nuevo desde backup. No lo trates como un failover stateless aleatorio.

**Minimum requirements for three-node cluster mode:**

- Fix one EMQX instance on each Swarm node.
- Usa un node name fijo para cada nodo EMQX. El node name no debe cambiar cuando se recrea el contenedor.
- Open cluster communication ports only between the three Swarm nodes.
- Expose the Dashboard only to the O&M network segment.
- El acceso MQTT puede exponerse mediante `APP VIP`, un MQTT VIP separado o un balanceador de capa 4. Confirma el método de entrada antes de implementar.
- No coloques el directorio de datos principal de EMQX en NFS. En modo fijo de un solo nodo, usa el disco local del nodo fijo y realiza backup correctamente. En modo de tres nodos, depende del mecanismo de clúster de EMQX.

### NFS/NAS Shared Storage

- NFS/NAS is an optional external dependency and does not count as one of the six VMs.
- Mount the same shared directory on all three Swarm nodes.
- Use it only for shared files that Tier0 Enterprise needs to access across nodes.
- Do not use it for PostgreSQL/TimescaleDB data directories.
- Do not use it for EMQX core runtime data.
- Configura una comprobación de montaje antes del arranque para evitar escribir en un directorio local después de que un servicio se mueva a un nodo donde NFS no está montado.
- Si NFS falla, el acceso de lectura/escritura a la base de datos no se afecta directamente, pero funciones como uploads, imports, exports y reports pueden fallar.

### APP VIP

- Deploy `APP VIP` on the three Swarm nodes. Use `keepalived + HAProxy` or `keepalived + Nginx`.
- The VIP is only the unified entry. It does not require a separate VM.
- Health checks should use the application `/health` or `/ready` endpoint.
- If any Swarm node fails, the VIP moves to another healthy node.

### PostgreSQL + TimescaleDB

**Recommended baseline:**

- PostgreSQL: `16` or a version certified on site.
- TimescaleDB: choose a stable version that matches the PostgreSQL major version.
- Primary-standby replication: physical streaming replication.
- Failover: `repmgrd + witness`.
- Database entry: `DB VIP`.
- Connection pool: preferably use `PgBouncer` on the application side or in a separate container to avoid too many database connections.

### Witness

- El nodo `Witness` solo almacena metadata de `repmgr witness` y ayuda en la detección y arbitraje de fallos de base de datos.
- `Witness` no almacena datos de negocio, no maneja tráfico de lectura/escritura de negocio y no expone acceso a base de datos para las aplicaciones.
- `Witness` needs stable network access. It does not require high storage performance.
- A `Witness` failure should not stop current business access, but it lowers the reliability of automatic failover.
- After `Witness` recovers, verify that it has rejoined the database cluster state.

## High Availability Capabilities and Failure Boundaries

<table>
  <thead>
    <tr>
      <th>Failure Scenario</th>
      <th>Expected Behavior</th>
      <th>Business Impact</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>A single Swarm node fails and affects the Tier0 Enterprise single replica</td>
      <td>Swarm reschedules the single replica to another healthy node. APP VIP continues to point to a healthy entry.</td>
      <td>Seconds to minutes of interruption, depending on image pulling and health checks.</td>
    </tr>
    <tr>
      <td>The EMQX node fails in single-node fixed mode</td>
      <td>Restore the original node first. If it cannot be restored, recover to a new node from backup.</td>
      <td>The messaging layer is interrupted. It cannot fail over without data loss like a stateless service.</td>
    </tr>
    <tr>
      <td>One node fails in the three-node EMQX cluster</td>
      <td>The remaining EMQX nodes continue to serve traffic. The failed node rejoins after recovery.</td>
      <td>Connections reconnect, and capacity is reduced.</td>
    </tr>
    <tr>
      <td>NFS/NAS fails</td>
      <td>Functions that depend on shared files fail. The database and business processes already in memory may not be interrupted.</td>
      <td>File-related functions such as uploads, imports, exports, and reports are affected.</td>
    </tr>
    <tr>
      <td>Two Swarm nodes fail</td>
      <td>The Swarm managers lose quorum. Running containers may continue, but reliable scheduling is not available.</td>
      <td>High risk. Manual recovery is required.</td>
    </tr>
    <tr>
      <td>The current primary database fails</td>
      <td>The standby database is promoted after Witness arbitration, and DB VIP fails over.</td>
      <td>Seconds to minutes of write interruption. The application must reconnect.</td>
    </tr>
    <tr>
      <td>The standby database fails</td>
      <td>The primary database continues to serve traffic, and a replication interruption alert is generated.</td>
      <td>No direct business interruption, but the database hot standby is lost.</td>
    </tr>
    <tr>
      <td>Witness fails</td>

## Backup, Monitoring, and Inspection

**Minimum delivery requirements:**

- Perform a full database backup every day and keep backups for at least `7` days.
- Enable WAL archiving or incremental backups to meet the on-site RPO.
- Perform a restore drill every month to verify that backups are usable.
- Monitor Swarm service health checks, container restart counts, node disk usage, memory, and CPU.
- Monitorea conexiones EMQX, suscripciones, throughput de mensajes, backlog de colas, fallos del motor de reglas y uso de directorios persistentes.
- Monitor NFS availability, mount status, capacity, inode usage, and read/write latency.
- Monitor PostgreSQL primary-standby delay, replication slots, WAL backlog, connections, slow SQL, and database bloat.
- Include the current VIP owner, repmgr cluster status, and Witness status in routine inspections.

## Implementation Sequence

1. Create 6 VMs and fix hostnames, IP addresses, time synchronization, and SSH access.
2. Initialize system baselines, firewall rules, disk mounts, and kernel parameters.
3. If NFS/NAS is used, first complete shared directory setup, permissions, mounting, and restart auto-mount verification.
4. Deploy the three-node Docker Swarm.
5. Deploy `APP VIP + HAProxy/Nginx`.
6. Deploy EMQX in either single-node fixed mode or three-node cluster mode.
7. Deploy PostgreSQL + TimescaleDB primary-standby.
8. Deploy repmgr witness, repmgrd, and the DB VIP failover script.
9. Publish the Tier0 Enterprise single-replica stack.
10. Verify connectivity, NFS read/write access, EMQX recovery, database failover, backup restore, and routine inspection.
11. Finalize operation documents, archive account credentials, and set up routine inspection tasks.

## Risks and Items to Confirm

- The actual on-site network segment, gateway, DNS, NTP, certificates, and image registry address must be confirmed.
- It is not yet confirmed whether Tier0 Enterprise needs local file persistence. If it does, add a shared storage or object storage solution.
- Debe confirmarse si NFS/NAS lo proporciona almacenamiento existente en sitio. Si NFS es un punto único de fallo, las funciones de archivos compartidos también tendrán un punto único de fallo.
- Cada nodo Swarm tiene solo un disco de sistema único de `100G`. Controla la cantidad de imágenes, logs de contenedores y archivos temporales. No es adecuado para almacenamiento de datos de negocio a largo plazo.
- `EMQX` es un servicio con estado. Antes de implementar, confirma si se usará modo fijo de un solo nodo o modo de clúster de tres nodos. En modo fijo de un solo nodo, la capa de mensajería tiene un punto único de fallo.
- El RPO/RTO de base de datos no se ha confirmado. El plan actual usa failover automático primary-standby y no puede prometer pérdida cero de datos.
- The `DB VIP` failover script must be verified on site, especially whether the virtualization platform allows VRRP.
- If the on-site network does not allow VRRP, use a load balancer, DNS switch, or virtualization-platform floating IP instead.
