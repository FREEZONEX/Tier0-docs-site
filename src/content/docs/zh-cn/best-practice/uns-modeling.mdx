## Demo UNS 如何组织
:::tip[What needs to be done first]
Before modeling data, answer the following questions.
- 哪些数据会从 UNS 被消费？
- 哪些数据会发布回 UNS？
:::


### 物理数据

来自机器和设备、并具有物理位置的工业数据，通常基于 ISA-95 标准建模。

#### ISA-95 设备层级
ISA-95 设备层级描述了组织内物理资产和控制系统的结构方式。
It is broken into the following levels:
- **Enterprise**：最高层级，代表整个组织，包括整体业务流程和决策实体，通常由 ERP 系统支持。  
- **Site**：代表特定地理位置或设施。每个 site 可以拥有自己的管理系统和本地生产控制。 
- **Area**：site 内的运营分区，通常是包装、装配或仓储等功能区域。 
- **Process Cell**：area 内的制造基本单元，包含生产特定产品或执行特定功能所需的全部设备。 
- **Unit**：process cell 中的独立部分，负责混合、加热或装配等特定功能。 
- **Equipment Module**：unit 内执行特定任务的模块化设备。例如在混合单元中，该模块可以是加热元件。  
- **Control Module**：最低层级，代表传感器、泵、执行器或 PID 控制回路等单个设备或控制器，用于直接控制物理过程。

### 业务数据

对于 ERP 中的生产订单、工单计划等业务计划和流程对象，应根据业务上下文进行建模。
:::tip[How to recognize business data]
Data that does not have a physical location.
:::
#### 业务数据 Modeling Guide
```
Factory / Business Domain / Business Object / Topic Type / Topic
```
- **Business Domain**：确认数据所属的业务流程或系统。
    ```
    ERP
    MES
    WMS
    QMS
    EAM
    LIMS
    Planning
    Inventory
    Maintenance
    Quality
    Logistics
    ...
    ```
- **Business Object**：业务域中被管理的核心对象。
    ```
    ProductionOrders
    WorkOrderPlan
    MaterialInventory
    QualityInspection
    MaintenanceOrders
    PurchaseOrders
    Shipments
    ...
    ```
- **Topic Type**：从 **Metric**、**Action** 和 **State** 中选择数据类型。

    | Topic Type | Used for | Examples |
    | --- | --- | --- |
    | STATE | 当前状态或当前事实 | 工单状态、订单状态、设备状态 |
    | METRIC | 连续变化的数值 | 温度、速度、产量、OEE、能耗 |
    | ACTION | 给外部执行者的命令 | 启动设备、停止设备、重置报警、请求下发工单 |

## 如何避免 Topic Explosion
Tier0 uses 3 fixed topic types: `Metric`, `Action`, and `State` to avoid topic explosion. 

- **将 `Metric` 用于高频实时数据**

  将高频测量值和遥测数据放在 `Metric` 下，而不是为每个信号细节继续扩展 namespace。

  <Tabs syncKey="topic-explosion-metric">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Machine_001
          ├── Temperature_Value
          ├── Temperature_Unit
          ├── Temperature_Status
          └── Speed_Value
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Metric
          └── Machine_001
  ```

  Model fields:

  ```json
  {
    "temperature": 80,
    "temperature_unit": "Celsius",
    "temperature_status": "Normal",
    "speed": 300
  }
  ```

    </TabItem>
  </Tabs>

- **将 `State` 用于低频状态数据**

  将低频状态、快照和当前业务状态放在 `State` 下。避免为作业、批次或订单等变化实体分别创建新 topic。

  <Tabs syncKey="topic-explosion-state">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── State
          ├── Job_001
          ├── Job_002
          └── Job_003
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── State
          └── Machine_001
  ```

  Model fields:

  ```json
  {
    "current_job": "Job_003",
    "product": "Product_A"
  }
  ```

    </TabItem>
  </Tabs>

- **将 `Action` 用于低频动作数据**

  将命令、请求和操作员动作放在 `Action` 下。动作输入应作为 topic 的字段，而不是为每个参数创建一个 topic。

  <Tabs syncKey="topic-explosion-action">
    <TabItem label="Bad">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Machine_001
          ├── Start_Command
          ├── Target_Height
          └── Light_On
  ```

    </TabItem>
    <TabItem label="Better">

  ```text
  Factory_A
  └── SMT_Line_1
      └── Action
          └── StartJob
  ```

  Model fields:

  ```json
  {
    "target_height": 12.5,
    "light_on": true,
    "requested_by": "operator"
  }
  ```

    </TabItem>
  </Tabs>

## Payload 结构

设计 topic payload 时，需要同时考虑要消费的数据和要回写的数据。

- **Action**

  Action payload 会自动包含 `_timestamp`，用于记录数据存入 Tier0 数据库的时间。

  <Tabs syncKey="payload-action">

  ```json

    "light_turn_on": true,
    "height_adjust": 12.5,
    "_timestamp": "2026-07-10T10:00:00Z"
  }
  ```

    </TabItem>
    <TabItem label="stop_job">

  {
