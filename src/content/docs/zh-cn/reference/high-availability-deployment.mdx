## 方案目标

该方案用于在离线或私有化环境中部署一套 Tier0 Enterprise。基础规模为 6 台虚拟机：

- 3 台应用节点组成 `Docker Swarm` 集群，运行单副本 Tier0 Enterprise 服务和有状态 **EMQX** 服务。
- 2 台数据库节点组成 `PostgreSQL + TimescaleDB` 主备集群。
- 1 台 `Witness` 节点参与数据库监控、仲裁和故障切换。

:::note[可选]
可增加外部 `NFS/NAS` 共享存储，并挂载到三台 Swarm 节点。它用于 Tier0 Enterprise 跨节点共享文件，是外部依赖，不计入本方案的 6 台业务 VM。
:::

:::tip[数据库资源变化时]
本方案假设每台数据库 VM 为 `4C / 8G / 500G`。如果需求是整个数据库集群合计 `4C / 8G / 500G`，实施前必须重新评估性能和容量。
:::

## 整体拓扑

<img class="t0-topology" src="/images/enterprise-ha-topology.png" alt="tier0-enterprise 六虚拟机南北向数据链路" />

图中仅展示核心链路：

- 业务访问路径：用户 / 现场系统 -> `APP VIP` -> 三台 `Docker Swarm` 节点 -> `DB VIP` -> 当前主库。
- `APP VIP` 将流量分发到三台 Swarm 节点，Swarm 节点通过 `DB VIP` 访问数据库。
- `APP VIP` 可在三台 Swarm 节点之间切换；`DB VIP` 跟随当前 PostgreSQL Primary，可在两台数据库节点之间切换。
- `Witness` 只参与数据库仲裁、主库选择和健康检查，不在业务访问路径中。
- `EMQX` 是有状态服务，默认应固定到一个节点并使用持久化存储；如果消息层也要求高可用，请部署三节点 EMQX 集群。

## 数据库监控与故障切换

| 阶段 | 链路 | 说明 |
| --- | --- | --- |
| 正常访问 | Swarm 节点上的 Tier0 Enterprise 副本 -> `DB VIP` -> 当前主库 | 应用始终只连接 `10.60.20.10:5432`。 |
| 正常复制 | 当前主库 -> 当前备库 | 通过 PostgreSQL WAL streaming replication 同步数据。 |
| 正常监控 | `DB-01` / `DB-02` -> `Witness` | `repmgrd` 上报状态，`Witness` 参与仲裁。 |
| 主库故障 | 备库 -> `Witness` | 备库请求仲裁，避免 split-brain。 |
| 备库提升 | 备库 `promote` | `Witness` 确认后，备库提升为新的主库。 |
| VIP 切换 | `DB VIP` -> 新主库 | `10.60.20.10` 绑定到新的主库。 |
| 应用恢复 | Tier0 Enterprise -> `DB VIP` -> 新主库 | 应用连接池重连，读写恢复。 |

关键边界：

- 业务服务只连接 `DB VIP:5432`，不允许直连 `ent-db-01/02`。
- `Witness` 不存储业务数据，不处理业务读写，也不向应用暴露数据库访问。
- `TimescaleDB` 作为 PostgreSQL extension 部署在两台数据库节点上，不拆成独立数据库实例。
- 推荐设计为 `PostgreSQL streaming replication + repmgr + repmgr witness + keepalived DB VIP`。
- 自动切换只处理主库不可用。原主库恢复后应先作为 standby 重新加入，不应自动抢回 primary。

## 部署建议

- 使用 `docker stack deploy` 部署 Tier0 Enterprise。
- 使用 `Docker config` 管理配置文件，使用 `Docker secret` 管理密码、数据库凭据和证书私钥。
- 使用现场内部镜像仓库，避免部署时依赖公网。
- 将 `APP VIP` 部署在三台 Swarm 节点上，可采用 `keepalived + HAProxy` 或 `keepalived + Nginx`。
- 数据库入口统一使用 `DB VIP`。应用侧建议使用连接池，避免数据库连接数过高。
- NFS/NAS 只用于 Tier0 Enterprise 跨节点共享文件，不要用于 PostgreSQL/TimescaleDB 数据目录，也不要用于 EMQX 核心运行时数据。

## 端口与防火墙

完整端口请参考 [标准端口清单](../standard-port-list/)。

默认不要开放以下端口：

- Docker 远程 API `2375` / `2376`，除非明确受控且只允许运维网段。
- 数据库节点 `80/443`，数据库节点不承载 Web 入口。
- Witness 对应用侧的 `5432` 访问，Witness 不处理业务 SQL。

## 高可用能力与故障边界

| 故障场景 | 系统行为 | 边界 |
| --- | --- | --- |
| 当前 Swarm 节点故障 | Swarm 将单副本服务重调度到其他健康节点。 | 需要等待容器重启和应用连接恢复。 |
| 单节点固定模式下 EMQX 节点故障 | 优先恢复原节点，无法恢复时从备份恢复到新节点。 | 消息层会中断，不能像无状态服务一样无损切换。 |
| EMQX 三节点集群中单节点故障 | 剩余节点继续服务，故障节点恢复后重新加入。 | 需要提前完成 EMQX 集群化配置。 |
| 当前主库故障 | 备库经 Witness 仲裁后提升，DB VIP 切换。 | 写入会中断数秒到数分钟，应用必须重连。 |
| 备库故障 | 主库继续服务，并产生复制中断告警。 | 自动故障切换能力下降。 |
| Witness 故障 | 当前主库继续服务。 | 自动切换能力下降，应优先恢复 Witness。 |
| NFS/NAS 故障 | 依赖共享文件的功能受影响。 | 数据库和已在内存中的业务流程不一定中断。 |

## 备份、监控与巡检

- 每天执行数据库备份，并在重大变更前强制备份。
- 监控 APP VIP、DB VIP、PostgreSQL replication lag、Swarm 节点状态、EMQX 状态和磁盘使用率。
- 定期演练主库切换、应用节点故障、备份恢复和共享存储故障。
- 上线前确认 SSH 免密、NTP 时间同步、镜像仓库、证书、DNS、网关和防火墙策略。

## 实施顺序

1. 确认现场网络、VIP、DNS、NTP、证书、镜像仓库和防火墙策略。
2. 准备 6 台 VM 的操作系统、磁盘、时间同步和基础安全配置。
3. 如果使用 NFS/NAS，先完成共享目录、权限、挂载和开机自动挂载验证。
4. 部署三节点 Docker Swarm。
5. 部署 `APP VIP + HAProxy/Nginx`。
6. 按需部署单节点固定 EMQX 或三节点 EMQX 集群。
7. 部署 PostgreSQL + TimescaleDB 主备。
8. 部署 repmgr witness、repmgrd 和 DB VIP 切换脚本。
9. 发布 Tier0 Enterprise 单副本 stack。

## 风险与待确认项

- 必须确认现场网段、网关、DNS、NTP、证书和镜像仓库地址。
- 需要确认 Tier0 Enterprise 是否需要本地文件持久化；如果需要，应增加共享存储或对象存储方案。
- `DB VIP` 切换脚本必须在现场验证，尤其要确认虚拟化平台是否允许 VRRP。
- 如果现场网络不允许 VRRP，请改用负载均衡、DNS 切换或虚拟化平台浮动 IP。
